# 매입 사이클 관리 설계안 (Purchase Cycle Management)

> 목표(사장님): 단순한 3축 대사표가 아니라, **매입 업무 전체 흐름(Purchase Cycle)** 을 관리한다.
> ```
> ERP 주문 → 세금계산서 수령 → 통장 지급 → 완료
> ```
> 이 흐름을 거래처 중심으로, "지금 어디까지 진행됐는가"와 "예외가 무엇인가"를 한눈에 보이게 한다.

작성일: 2026-07-08 · v2 (ChatGPT 피드백 반영) · 상태: **설계안 (구현 전 승인 대기)**

---

## 0. 관점 전환 (v1 → v2)

| | v1 (3축 대사) | v2 (매입 사이클) |
|---|---|---|
| 중심 | 숫자 대사표 | **업무 흐름 + 진행 상태** |
| 사용자 질문 | "금액이 맞나?" | **"다온글로비스는 지금 어디까지 됐지?"** |
| 우선 화면 | 거래처×기간 대사 | **예외 관리 + 거래처별 진행상태** |
| 대사 결과 | "확인함"(고정) | **"재검토 필요"(데이터 바뀌면 자동)** |

숫자 대사는 이 흐름을 떠받치는 **하부 계산**으로 두고, 화면은 흐름·상태·예외 중심으로 만든다.

---

## 1. 유지하는 원칙 (피드백 1·2 — 그대로)

- **건별 자동매칭은 하지 않는다.** ERP=품목단위 / 계산서=거래처월합계 / 통장=분할지급 → 1:1 불가.
  거래처 × 기간 집계로 대사한다. (실데이터: 양쪽 있는 460조합 중 금액 근사 36개뿐)
- **역할 분리 유지** (회계정책 §3): ERP=매출원가 근거 · 세금계산서=비용 발생 · 통장=미지급금 상계.
  이 구조가 이중계상을 막는다. 사이클 관리는 **추적을 더할 뿐 회계 구조를 바꾸지 않는다.**

---

## 2. 매입 사이클 상태 모델 (피드백 4 — 핵심 신규)

거래처별로 세 축의 금액·건수를 집계해 **상태를 자동 판정**한다. 사용자는 숫자를 해석하지 않고
상태만 보고 업무를 파악한다.

| 상태 | 판정 (제안) | 의미 |
|---|---|---|
| **계산서 대기** | ERP 매입 있음 & 세금계산서 없음/부족 | 물건은 받기로 했는데 계산서 미수령 |
| **지급 대기** | 세금계산서 있음 & 지급(통장 연결) < 계산서 | 계산서 받았고 아직 안 갚음 → 미지급 |
| **금액 차이** | ERP ↔ 계산서 차이가 허용범위 초과(예 >10%) | 단가·수량·시차·누락 의심 → 확인 필요 |
| **완료** | ERP ≈ 계산서 ≈ 지급 (허용범위 내) | 정상 종결 |
| **경비성(ERP 무관)** | 계산서만 있고 ERP 매입 없음 | 사무·임차·수수료 등 — 사이클 대상 아님(정상) |

### 실데이터 시뮬레이션 (327개 매입처, 지금 기준)
- 경비성(ERP 없음): **268곳** — 정상 유형(예외 아님). ERP 주문과 무관한 판관비성 매입
- 지급 대기: **27곳** (예: 두일지엘 계산서 19.9M / 지급 0)
- 금액 차이 큼: **24곳** (예: 루체컴퍼니 ERP 28.6M / 계산서 69.7M)
- 근사 완료: 8곳
- → **"지급 대기 + 금액 차이" 51곳이 곧 예외 관리 대상**

> 보수적 주의: ERP 매입가와 세금계산서 공급가는 **시차(월 걸침)·부분 계산서·현금매입** 때문에
> 구조적으로 차이날 수 있다. 그래서 "금액 차이"는 **틀림이 아니라 확인 요청** 신호다.
> 허용범위(%)·시차 보정(전월/익월 합산)은 구현 시 실데이터로 조정한다.

## 3. 거래처 진행상태 화면 (피드백 3·5 — 거래처 중심 + 흐름 시각화)

거래처를 **먼저 선택**하고 기간은 필터로. 선택하면 흐름을 파이프라인으로:

```
[다온글로비스]                                        기간 필터: 2025 전체 ▾

 ERP 주문 ────▶ 세금계산서 ────▶ 지급 ────▶ 완료
  473.7M          348.9M          340.0M      [지급 대기]
  (품목 N건)      (계산서 M건)     (지급 K건)

 ● 미지급 잔액 8.9M   ● ERP-계산서 차이 124.8M(시차 의심)
```

- 각 단계 = 클릭 가능 → 해당 건 목록 → 원본 드릴다운 (기존 추적성 재사용)
- 단계 사이 화살표에 **진행/막힘 표시** (예: 계산서→지급이 막혀 있으면 강조)
- 상단에 현재 **상태 배지** 1개 (계산서 대기 / 지급 대기 / 금액 차이 / 완료)

---

## 4. 예외 관리 화면 (피드백 7 — 가장 중요, 최우선 구현)

정상보다 **예외를 먼저** 보여준다. 진입 시 기본 탭 = 예외.

| 예외 유형 | 조건 | 업무 의미 |
|---|---|---|
| 계산서 미수령 | ERP 매입 있음, 계산서 없음/부족 | 세금계산서 요청해야 함 |
| 지급 대기(미지급) | 계산서 있음, 지급 안 됨 | 대금 지급 예정 |
| 금액 차이 큼 | ERP ↔ 계산서 차이 초과 | 단가·수량·누락 확인 |
| 과다 지급 | 지급액 > 계산서 합계 | 이중지급·선급금 의심 (우선 경고) |

- 금액·경과일 순 정렬 → 큰 것부터 처리
- 각 행에서 바로 거래처 진행상태 화면으로 이동
- 이 화면이 사실상 **매입 담당자의 To-Do 리스트**가 된다

---

## 5. 재검토 상태 (피드백 6 — 잠금 아님)

- 대사 결과는 **잠그지 않는다.** 회계 Confirm처럼 고정하지 않는다.
- 사용자가 "확인함"을 눌러도, 이후 그 거래처에 **새 세금계산서/지급이 들어오면 자동으로
  "재검토 필요"로 전환**된다.
- 확인 이력은 남긴다: 누가·언제·그 시점 금액 스냅샷. 다음 확인 때 "무엇이 바뀌었는지" 비교.
- 구현: 거래처별 `last_reviewed_at` + 스냅샷(ERP/계산서/지급 합계). 현재 합계가 스냅샷과
  달라지면 "재검토 필요" 자동 표시. (별도 잠금 플래그 없음)

---

## 6. 자동 / 수동 경계 (정책 §4)

| 작업 | 자동 | 사용자 |
|---|---|---|
| 상태 판정·예외 추출·차이 계산 | 전부 자동 | — |
| ②↔③ 지급 후보 추천 | 금액·거래처·기간으로 추천 | 매칭 확정 |
| "확인함" / 재검토 | 새 데이터 시 자동 재검토 전환 | 확인 행위 |
| 계정·분개 | 확정 시 자동 생성(기존) | 계정 확정 |

시스템은 어떤 경우에도 "같다/완료다"를 **임의 확정하지 않는다.**

---

## 7. 구현 로드맵 (단계별 검증 보고서 포함)

1. **매입 사이클 상태 엔진** (거래처별 ERP/계산서/지급 집계 + 상태 판정 RPC/API)
2. **예외 관리 화면** (최우선 — 계산서 미수령/지급 대기/금액 차이/과다 지급)
3. **거래처 진행상태 화면** (파이프라인 시각화 + 드릴다운, 거래처 중심 조회)
4. **②↔③ 지급 자동매칭 확장** (미매칭 계산서에 통장 지급 후보 추천 — 수금매칭 로직 재사용)
5. **재검토 상태 + 확인 이력** (스냅샷 비교, 잠금 없음)
6. 실데이터 검증 — 예외 51곳부터 원인 규명, 더존 대사와 연계

- 회계정책 §3(역할분리)·§4(추천만)·§6(추적성) 전부 준수.
