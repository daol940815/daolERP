# 근태 관리 시스템 기획서 (초안)

| 항목 | 내용 |
|---|---|
| 문서 버전 | v0.1 (초안) |
| 작성일 | 2026-07-07 |
| 상태 | 검토 대기 |
| 프로젝트 | daolERP — 근태 관리 모듈 |

---

## 1. 개요

### 1.1 배경 및 목적
- 출퇴근 기록, 연차/휴가 관리, 초과근무 관리 등 근태 업무를 수기(엑셀, 메신저 결재 등)로 처리하면서 발생하는 누락·오류·이력 관리의 어려움을 해소한다.
- 우선 **독립 실행 가능한 근태 관리 시스템**으로 개발·운영하고, 실사용 검증 후 문제가 없으면 **daolERP 본체에 모듈로 병합**한다.
- 연차 발생/소진 계산을 자동화하여 근로기준법 준수와 인사 담당자의 업무 부담 경감을 동시에 달성한다.

### 1.2 개발 전략 (2단계)
| 단계 | 내용 |
|---|---|
| 1단계 (독립 운영) | 근태 관리 단독 시스템으로 개발. 자체 로그인/사용자 관리 포함. 실무에서 사용하며 검증 |
| 2단계 (ERP 병합) | 검증 완료 후 daolERP의 한 모듈로 병합. 사용자/조직/권한은 ERP 공통 모듈로 이관 |

> **병합을 전제로 한 설계 원칙**: 사용자·조직·권한 등 공통 도메인은 근태 도메인과 코드/DB 레벨에서 분리해 두어, 병합 시 근태 도메인 코드는 최소 수정으로 이식 가능하도록 한다. (상세: 9장)

### 1.3 용어 정의
| 용어 | 정의 |
|---|---|
| 근태 | 출근, 퇴근, 지각, 조퇴, 결근, 외근, 재택 등 근무 상태 전반 |
| 연차 | 근로기준법상 연차유급휴가 |
| 반차 | 연차 0.5일 단위 사용 (오전/오후) |
| 대체휴가 | 휴일근무에 대한 보상 휴가 |
| 근태 마감 | 월 단위로 근태 기록을 확정하여 급여 계산 등에 사용하는 것 |

## 2. 범위

### 2.1 포함 범위 (1단계)
1. 출퇴근 기록 (웹 기반 체크인/체크아웃, 수동 보정 신청)
2. 연차/휴가 관리 (자동 발생, 신청, 승인, 잔여 관리, 촉진 알림)
3. 초과근무/휴일근무 신청 및 승인
4. 근태 현황 조회 및 월 마감
5. 통계/리포트 (개인별·부서별·기간별)
6. 관리자 기능 (사용자/조직/휴가 정책/공휴일 관리)
7. 알림 (승인 요청/결과, 연차 촉진 등)

### 2.2 제외 범위 (추후 검토)
- 급여 계산 연동 (마감 데이터 내보내기까지만 제공)
- 출입 통제 장비(카드/지문) 연동 — 2단계 이후 검토
- 모바일 네이티브 앱 (반응형 웹으로 대체)
- 유연근무제(선택근로/재량근로)의 복잡한 정산 — 기본 근무제 우선

## 3. 사용자 및 권한

| 역할 | 설명 | 주요 권한 |
|---|---|---|
| 일반 직원 | 전 임직원 | 본인 출퇴근 기록, 휴가/초과근무 신청, 본인 현황 조회 |
| 승인자 (팀장/부서장) | 결재 라인 상의 승인 권한자 | 소속 부서원 신청 승인/반려, 부서 근태 현황 조회 |
| 인사 담당자 (HR) | 인사팀 | 전사 근태 조회, 근태 보정, 휴가 정책 설정, 월 마감, 리포트 |
| 시스템 관리자 | IT 담당 | 사용자/조직/권한 관리, 시스템 설정 |

- 역할은 중복 부여 가능 (예: 팀장은 일반 직원 + 승인자).
- 승인 라인은 기본 1단계(직속 부서장)로 시작하되, 다단계 결재로 확장 가능한 구조로 설계한다.

## 4. 기능 요구사항

### 4.1 출퇴근 관리
| ID | 기능 | 설명 | 우선순위 |
|---|---|---|---|
| ATT-01 | 출근/퇴근 체크 | 웹 화면에서 버튼 클릭으로 기록. 서버 시간 기준 저장 | 필수 |
| ATT-02 | 근무 상태 구분 | 정상 / 지각 / 조퇴 / 결근 / 외근 / 재택 / 출장 자동·수동 분류 | 필수 |
| ATT-03 | 기록 보정 신청 | 체크 누락·오류 시 사유와 함께 보정 신청 → 승인자 승인 후 반영. 원본 기록은 보존하고 보정 이력 별도 관리 | 필수 |
| ATT-04 | 오늘의 근태 현황 | 본인 출퇴근 시각, 근무 시간, 주간 누적 근무시간 표시 | 필수 |
| ATT-05 | IP/위치 제한 | 사내 IP 대역에서만 체크 허용 (설정으로 on/off) | 선택 |

### 4.2 연차/휴가 관리

#### 4.2.1 휴가 유형
| 유형 | 단위 | 유급 여부 | 비고 |
|---|---|---|---|
| 연차 | 1일 / 0.5일(반차) | 유급 | 근로기준법 기준 자동 발생 |
| 대체휴가 | 1일 / 0.5일 | 유급 | 휴일근무 승인 건에서 발생 |
| 경조휴가 | 일 단위 | 유급 | 유형별 일수 정책 설정 (결혼, 조사 등) |
| 병가 | 일 단위 | 정책 설정 | 회사 규정에 따름 |
| 공가 | 일 단위 | 유급 | 예비군, 건강검진 등 |
| 무급휴가 | 일 단위 | 무급 | |

> 휴가 유형은 HR이 관리 화면에서 추가/수정 가능한 **정책 데이터**로 설계한다 (코드 하드코딩 금지).

#### 4.2.2 연차 자동 발생 규칙 (근로기준법 제60조 기준)
| 근속 기간 | 발생 규칙 |
|---|---|
| 1년 미만 | 1개월 개근 시 1일 (최대 11일) |
| 1년 이상 | 1년간 80% 이상 출근 시 15일 |
| 3년 이상 | 최초 1년 초과 후 매 2년마다 1일 가산 (15+1, 16... 최대 25일) |

- 회계연도 기준 부여 / 입사일 기준 부여 중 **회사 정책으로 선택** 가능하게 설계 (기본값: 입사일 기준). *→ 확정 필요 (11장 참조)*
- 연차 사용기한(발생일로부터 1년) 및 이월 정책 설정 지원.
- 연차 사용 촉진: 사용기한 임박 시 본인·HR에게 알림 (1단계는 알림까지, 법정 촉진 절차 서면 통보는 추후).

#### 4.2.3 휴가 신청/승인
| ID | 기능 | 설명 | 우선순위 |
|---|---|---|---|
| LEV-01 | 휴가 신청 | 유형, 기간(시작~종료), 반차 여부, 사유 입력. 잔여일수 초과 시 신청 불가 | 필수 |
| LEV-02 | 승인/반려 | 승인자가 승인 또는 반려(사유 필수). 결과 알림 발송 | 필수 |
| LEV-03 | 신청 취소 | 승인 전 자유 취소, 승인 후 시작일 전이면 취소 신청(승인 필요) | 필수 |
| LEV-04 | 잔여 연차 조회 | 발생/사용/잔여 내역을 발생 건별로 조회 | 필수 |
| LEV-05 | 부서 휴가 캘린더 | 부서원 휴가 일정을 월 캘린더로 표시 (승인자 판단 지원) | 필수 |
| LEV-06 | 연차 수동 조정 | HR이 발생/차감 수동 조정 (사유 필수, 이력 보존) | 필수 |

- 휴가 일수 계산 시 **주말·공휴일은 자동 제외**.
- 잔여 연차 차감은 승인 시점에 확정, 신청 시점에는 가차감(pending)으로 이중 신청 방지.

### 4.3 초과근무/휴일근무
| ID | 기능 | 설명 | 우선순위 |
|---|---|---|---|
| OT-01 | 초과근무 신청 | 사전 신청 원칙 (일시, 예상 시간, 사유). 사후 신청 허용 여부는 정책 설정 | 필수 |
| OT-02 | 승인/반려 | 승인자 승인. 승인된 건만 초과근무 시간으로 집계 | 필수 |
| OT-03 | 주 52시간 모니터링 | 주간 총 근무시간이 한도 임박/초과 시 본인·승인자·HR 알림 | 필수 |
| OT-04 | 휴일근무 → 대체휴가 | 휴일근무 승인 건에 대해 대체휴가 부여 (배율 정책 설정) | 선택 |

### 4.4 근태 마감
| ID | 기능 | 설명 | 우선순위 |
|---|---|---|---|
| CLS-01 | 월 마감 | HR이 월 단위 근태 확정. 마감 후 해당 월 기록 수정 불가 (마감 해제는 HR만 가능, 이력 기록) | 필수 |
| CLS-02 | 마감 전 검증 | 미처리 보정 신청, 승인 대기 건, 기록 누락자 목록 표시 | 필수 |
| CLS-03 | 마감 데이터 내보내기 | 급여 계산용 집계 데이터 Excel/CSV 다운로드 | 필수 |

### 4.5 통계/리포트
- 개인별 월간 근태 요약 (근무일수, 지각/조퇴/결근, 초과근무 시간, 휴가 사용)
- 부서별/전사 기간별 집계
- 연차 사용률 현황 (촉진 대상자 식별)
- Excel 다운로드 지원

### 4.6 관리자 기능
- 사용자 관리: 등록/수정/퇴사 처리 (퇴사자 데이터는 보존, 로그인만 차단)
- 조직 관리: 부서 트리, 부서장(승인자) 지정
- 정책 관리: 휴가 유형/일수, 연차 부여 기준, 근무시간(코어타임/출퇴근 기준 시각), 지각 판정 기준
- 공휴일 관리: 연도별 공휴일 등록 (대체공휴일 포함)

### 4.7 알림
| 이벤트 | 수신자 | 채널 (1단계) |
|---|---|---|
| 휴가/초과근무/보정 신청 | 승인자 | 시스템 내 알림 + 이메일 |
| 승인/반려 결과 | 신청자 | 시스템 내 알림 + 이메일 |
| 연차 사용기한 임박 (D-60/D-30) | 본인, HR | 시스템 내 알림 + 이메일 |
| 주 52시간 임박 | 본인, 승인자, HR | 시스템 내 알림 |

> 2단계(ERP 병합) 시 ERP 공통 알림 체계로 통합.

## 5. 비기능 요구사항

| 항목 | 요구사항 |
|---|---|
| 보안 | 비밀번호 해시 저장(bcrypt 등), 역할 기반 접근 제어(RBAC), 개인정보(근태 기록) 접근은 권한자만 |
| 감사 이력 | 근태 기록 보정, 연차 조정, 마감/마감해제 등 주요 변경은 변경자·일시·사유를 감사 로그로 보존 |
| 데이터 보존 | 근태 기록 3년 이상 보존 (근로기준법 관련 서류 보존 의무 고려) |
| 시간 기준 | 서버 시간(KST) 단일 기준. 클라이언트 시간 미신뢰 |
| 가용성 | 출퇴근 체크가 몰리는 시간대(08~10시, 18~19시) 응답 지연 없도록 |
| 사용성 | 반응형 웹 (모바일 브라우저에서 출퇴근 체크·휴가 신청 가능) |

## 6. 화면 구성 (초안)

```
[일반 직원]
├─ 대시보드: 출퇴근 체크 버튼, 오늘 근태, 주간 근무시간, 잔여 연차, 대기 중 신청
├─ 내 근태: 월 캘린더/목록 뷰, 보정 신청
├─ 휴가: 잔여 현황, 신청, 신청 내역, 부서 휴가 캘린더
├─ 초과근무: 신청, 내역
└─ 알림함

[승인자 추가 메뉴]
├─ 승인함: 대기 건 목록 → 승인/반려
└─ 부서 현황: 부서원 근태/휴가 현황

[HR 추가 메뉴]
├─ 전사 근태 현황
├─ 근태 마감
├─ 통계/리포트
├─ 정책 관리 (휴가 유형, 연차 기준, 근무시간, 공휴일)
└─ 연차 조정

[시스템 관리자]
└─ 사용자/조직/권한 관리
```

## 7. 데이터 모델 (초안)

```
■ 공통 도메인 (2단계에서 ERP 공통 모듈로 대체)
users            사용자 (로그인 계정)
employees        직원 (사번, 이름, 입사일, 퇴사일, 재직 상태)
departments      부서 (트리 구조, 부서장)
roles / user_roles  역할 및 매핑

■ 근태 도메인 (병합 시 그대로 이식)
attendance_records   출퇴근 기록 (직원, 일자, 출근/퇴근 시각, 상태, 원본/보정 구분)
attendance_corrections  보정 신청 (대상 기록, 변경 내용, 사유, 상태, 승인자)
work_policies        근무 정책 (기준 출퇴근 시각, 지각 기준 등)
holidays             공휴일

leave_types          휴가 유형 (정책 데이터)
leave_grants         연차/휴가 발생 (직원, 유형, 발생일, 일수, 사용기한, 발생 사유)
leave_requests       휴가 신청 (기간, 일수, 상태: 대기/승인/반려/취소)
leave_usages         발생 건별 사용 매핑 (grant ↔ request, 선입선출 차감)

overtime_requests    초과근무 신청
approvals            승인 이력 (신청 유형 공통, 다단계 확장 대비)
notifications        알림
audit_logs           감사 로그
monthly_closings     월 마감 상태
```

- 연차 잔여는 컬럼으로 저장하지 않고 `leave_grants − leave_usages`로 계산 (정합성 우선). 성능 이슈 시 캐시 도입.
- 모든 근태 도메인 테이블은 `users`가 아닌 `employees`를 참조 → ERP 병합 시 계정 체계가 바뀌어도 근태 데이터 영향 없음.

## 8. 기술 스택 (제안)

> daolERP 본체의 스택이 아직 확정되지 않았으므로, 아래는 제안이며 **ERP 본체 스택 결정과 함께 확정**해야 한다. 근태 시스템이 사실상 ERP의 첫 모듈이 되므로, 여기서 정한 스택이 ERP 표준이 될 가능성이 높다.

| 구분 | 제안 | 비고 |
|---|---|---|
| 백엔드 | Spring Boot (Java/Kotlin) 또는 NestJS (TypeScript) | 팀 숙련도에 따라 선택 |
| 프론트엔드 | React + TypeScript | 반응형 웹 |
| DB | PostgreSQL | 트랜잭션·정합성 중시 |
| 인증 | 1단계: 자체 세션/JWT → 2단계: ERP 통합 인증(SSO) | |
| 배치 | 스케줄러 (연차 자동 발생, 촉진 알림, 52시간 집계) | |
| 배포 | Docker 기반 | ERP 병합 시 동일 인프라 사용 |

## 9. ERP 병합 전략

1. **모듈 경계 유지**: 근태 도메인 코드는 `attendance` 패키지/모듈로 격리하고, 공통 도메인(사용자/조직/권한/알림)에는 인터페이스를 통해서만 접근한다.
2. **병합 절차 (2단계)**:
   - ERP 공통 모듈(사용자/조직/권한)이 준비되면 근태의 공통 도메인 구현체를 ERP 것으로 교체
   - 직원/부서 데이터 매핑 마이그레이션 (사번 기준)
   - 근태 도메인 테이블은 데이터 이관 없이 스키마째 이동 (동일 DB라면 그대로 사용)
   - 화면은 ERP 셸(공통 레이아웃/메뉴) 안에 근태 메뉴로 편입
3. **검증 기준 (병합 GO/NO-GO)**: 독립 운영 기간 중 아래 충족 시 병합 진행
   - 1회 이상 월 마감을 실데이터로 무리 없이 완료
   - 연차 자동 발생 결과가 수기 계산과 전수 일치
   - 사용자 불편 사항(버그/개선) 처리 완료
