## 1. 개요

### 1.1 배경 및 목적
- 출퇴근 기록, 연차/휴가 관리, 초과근무 관리 등 근태 업무를 수기(엑셀, 메신저 결재 등)로 처리하면서 발생하는 누락·오류·이력 관리의 어려움을 해소한다.
- 우선 **독립 실행 가능한 근태 관리 시스템**으로 개발·운영하고, 실사용 검증 후 문제가 없으면 **daolERP 본체에 모듈로 병합**한다.
- 연차 발생/소진 계산을 자동화하여 근로기준법 준수와 인사 담당자의 업무 부담 경감을 동시에 달성한다.
- 근태 모듈에서 정립한 설계 원칙(5.1장)을 향후 ERP 전체 모듈의 표준 아키텍처로 활용한다.

### 1.2 개발 전략 (2단계)
| 단계 | 내용 |
|---|---|
| 1단계 (독립 운영) | 근태 관리 단독 시스템으로 개발. 자체 로그인/사용자 관리 포함. 실무에서 사용하며 검증 |
| 2단계 (ERP 병합) | 검증 완료 후 daolERP의 한 모듈로 병합. 사용자/조직/권한은 ERP 공통 모듈로 이관 |

> **병합을 전제로 한 설계 원칙**: 사용자·조직·권한 등 공통 도메인은 근태 도메인과 코드/DB 레벨에서 분리해 두어, 병합 시 근태 도메인 코드는 최소 수정으로 이식 가능하도록 한다. (상세: 10장)

### 1.3 설계 철학 — 단일 법인 전용 시스템 (멀티테넌트 미채택)

본 시스템과 daolERP 전체는 **하나의 법인 전용 시스템**으로 설계한다.

- 여러 법인을 하나의 시스템에서 관리하는 **멀티테넌트 구조는 채택하지 않는다.**
- 향후 다른 법인에 적용할 경우 **완성된 시스템을 복제하여 각각 독립적으로 운영**하며, 각 법인은 DB, 사용자, 권한, 회계 데이터를 모두 독립적으로 관리한다.
- 이는 각 법인의 회계·경영 시스템을 완전히 분리 운영하기 위한 장기 전략이다.

```
다올커머스 ERP  (본 시스템 — 최초 구축·검증)
   ├─ 복제 → 올푸름 ERP   (독립 DB / 독립 배포 / 독립 운영)
   ├─ 복제 → 요아럽 ERP   (독립 DB / 독립 배포 / 독립 운영)
   └─ 복제 → ...
```

**설계에 미치는 영향**
- 전 테이블에 `company_id`를 두지 않는다. 하나의 DB에는 항상 하나의 법인 데이터만 존재한다.
- 법인 선택/전환 기능, 법인 간 권한 개념을 만들지 않는다.
- 법인별 고유 정보(회사명, 로고, 기본 정책, 메일 설정 등)는 하드코딩하지 않고 **시스템 설정(4.13장)**으로 분리하여, 복제 후 설정 변경만으로 새 법인에 적용 가능하게 한다.

> **복제 방식 권고**: 여기서 "복제"는 법인마다 코드를 포크(fork)하는 것이 아니라, **동일 코드베이스를 법인별로 독립 배포(독립 DB·인프라)**하는 방식을 권장한다. 코드를 포크하면 버그 수정과 기능 개선을 법인 수만큼 반복 적용해야 하므로, 코어 코드는 하나의 저장소로 유지하고 법인별 차이는 시스템 설정과 배포 구성으로 흡수한다. (미확정 사항 #12)

### 1.4 핵심 설계 구조

시스템의 근간이 되는 계산 흐름은 다음과 같다. 가운데의 **근태 계산 엔진(Attendance Engine)**이 모든 근태 판정을 담당하는 단일 지점이다. (상세: 5.2장)

```
근무정책 (Work Policy)            "이 직원은 어떤 규칙으로 일하는가"        [Master]
      │  배치 생성 + 개별 조정
      ▼
근무일정 (Work Schedule)          "오늘 몇 시부터 몇 시까지 근무 예정인가"   [Master]
      │
      ▼                          ┌ 입력: 정책 버전, 근무일정,
출퇴근 이벤트 (Attendance Events)  │       출퇴근 이벤트(보정 반영),        [Transaction]
      │                          │       승인된 휴가/초과근무
      ▼                          │
근태 계산 엔진 (Attendance Engine) ┤ 순수 계산 모듈 — DB 쓰기 없음,          [Engine]
      │                          │ 같은 입력 → 항상 같은 출력
      ▼                          └ 출력: 일별/기간별 근태 결과
근태 결과 (Attendance Result)      지각/조퇴/결근/근무시간/초과근무시간      [계산값 — 저장 안 함]
      │  월마감 시점에 확정
      ▼
마감 스냅샷 (Closing Snapshot)     급여 계산의 근거 — 마감 후 불변           [Snapshot]
```

- **원본(Raw) 데이터만 저장하고 파생값(근무시간, 지각시간 등)은 저장하지 않는다.** 정책 변경 시 재계산이 가능하고 정합성이 유지된다.
- **단, 월마감 시점에는 엔진의 계산 결과를 스냅샷으로 확정 저장한다.** 급여 계산의 근거는 이후 정책이 바뀌어도 변하지 않아야 하기 때문이다. (마감 전 = 항상 계산, 마감 후 = 스냅샷 조회)
- 이 구조는 ERP 공통 설계 원칙 `Master → Transaction → Engine → Snapshot`(5.1장)의 근태 구현체다.

### 1.5 용어 정의
| 용어 | 정의 |
|---|---|
| **User (사용자)** | 로그인 계정. 인증·권한의 주체 |
| **Employee (직원)** | 인사 객체. 근태·연차 등 모든 업무 데이터의 주체 |
| 근태 | 출근, 퇴근, 지각, 조퇴, 결근, 외근, 재택 등 근무 상태 전반 |
| 근무정책 | 근무 형태(고정/시차/자율), 기준 출퇴근 시각, 휴게시간, 지각 판정 기준 등의 규칙 묶음 |
| 근무일정 | 직원×일자 단위로 생성되는 근무 예정 정보 (근무일 여부, 예정 출퇴근 시각) |
| 시프트 | 교대근무 등에서 사용하는 근무 시간대 단위 (주간조/야간조 등) |
| 근태 계산 엔진 | 근무일정과 출퇴근 이벤트를 대조하여 근태를 판정하는 순수 계산 모듈 |
| 연차 | 근로기준법상 연차유급휴가 |
| 반차 | 연차 0.5일 단위 사용 (오전/오후) |
| 대체휴가 | 휴일근무에 대한 보상 휴가 |
| 근태 마감 | 월 단위로 근태 기록을 확정하여 급여 계산 등에 사용하는 것 |

## 2. 범위

### 2.1 포함 범위 (1단계)
1. 직원/조직 관리 (직급, 직책, 고용형태, 정책 연결, 변경 이력 포함)
2. 근무정책 관리 (직원별/부서별 상이한 정책 적용)
3. 근무일정 관리 (정책 기반 자동 생성 + 개별 조정)
4. 출퇴근 기록 (이벤트 단위 기록, 수동 보정 신청)
5. 근태 계산 엔진 (지각/조퇴/결근/근무시간 판정)
6. 연차/휴가 관리 (정책 기반 자동 발생, 신청, 승인, 잔여 관리, 촉진 알림)
7. 초과근무/휴일근무 신청 및 승인
8. 승인라인 관리 (다단계 결재)
9. 첨부파일 (병가 진단서, 출장계획서 등)
10. 근태 현황 조회 및 월 마감 (자동 검증 포함)
11. 통계/리포트 (개인별·부서별·기간별, HR 활용 리포트)
12. 관리자 기능 (사용자/조직/정책/휴일 관리)
13. 알림 (아웃박스 큐 기반, 채널 확장 가능 구조)
14. 공통 스케줄러 (배치 작업 등록/실행/이력 관리)
15. 감사 로그 (before/after, 변경 사유 필수)
16. 시스템 설정 (회사 정보, 기본 정책, 메일 — 법인 복제 시 변경 지점)
17. 관리자(HR) 대시보드 (당일 인원 현황, 승인 대기 등)
18. 퇴사 프로세스 (로그인 차단·일정 정리·승인라인 정리·연차 정산)
19. 데이터 Import/Export (Excel — 초기 데이터 일괄 등록)

### 2.2 제외 범위 (추후 검토)
- 급여 계산 연동 (마감 스냅샷 내보내기까지만 제공)
- 출입 통제 장비(카드/지문) 연동 — 2단계 이후 검토
- 모바일 네이티브 앱 (반응형 웹으로 대체)
- GPS 기반 위치 기록 — 개인정보 이슈 검토 후 결정 (12장 참조)
- 교대근무(시프트) 자동 편성 — 시프트 데이터 구조는 반영하되, 편성 자동화는 실수요 확인 후
- 범용 캘린더(Calendar/Calendar Events) 모듈 — 1단계는 `holidays`(유형·범위 포함)로 충분. ERP 병합 시 전사 일정(워크샵 등)까지 포괄하는 공통 캘린더로의 승격을 재검토 (holidays 데이터는 캘린더 이벤트로 이관 가능한 구조 유지)
- Slack/카카오워크/SMS 알림 채널 — 아웃박스 구조상 어댑터 추가로 확장 가능, 1단계 채널은 시스템 내 알림 + 이메일

## 3. 사용자 및 권한

### 3.1 User와 Employee의 분리 (ERP 공통 원칙)

**User(로그인 계정)와 Employee(인사 객체)는 별개의 개념**이며, 문서·코드 전반에서 혼용하지 않는다.

| 구분 | User | Employee |
|---|---|---|
| 정의 | 로그인 계정 (인증·권한의 주체) | 직원 (업무 데이터의 주체) |
| 근태 데이터 참조 | 참조하지 않음 | 모든 근태·연차 데이터가 참조 (사번 기준) |
| 퇴사 시 | 로그인만 비활성화 | 재직 상태 변경, 데이터는 보존 |

- 관계는 **1:0..1**이 기본이되 1:1을 강제하지 않는다:
  - 퇴사자 — Employee는 유지, User만 비활성화
  - 외부 사용자(노무사, 감사인 등) — User는 있으나 Employee가 아닐 수 있음
  - 시스템 계정(배치, 연동) — User만 존재
- 인증·권한은 User에, 업무 규칙(정책 배정, 승인라인, 근태 판정)은 Employee에 귀속된다. ERP 병합 시 교체되는 것은 User(계정 체계)이며 Employee와 업무 데이터는 영향받지 않는다.

### 3.2 역할
| 역할 | 설명 |
|---|---|
| 일반 직원 | 전 임직원. 본인 출퇴근 기록, 휴가/초과근무 신청, 본인 현황 조회 |
| 승인자 | 승인라인 상의 결재 권한자 (팀장/본부장/대표 등). 담당 신청 건 승인/반려, 소속 조직 현황 조회 |
| 인사 담당자 (HR) | 전사 근태 조회, 근태 보정, 정책 설정, 월 마감, 리포트 |
| 시스템 관리자 | 사용자/조직/권한 관리, 시스템 설정, 스케줄러/감사 로그 조회 |

### 3.3 권한 설계 — 기능 단위 권한 기반 RBAC
메뉴 접근 권한만으로는 부족하므로, **기능 단위 권한(permission)을 정의하고 역할(role)은 권한의 묶음**으로 설계한다.

| 권한 예시 | 설명 |
|---|---|
| `attendance.read.self / dept / all` | 근태 조회 범위 |
| `attendance.correct` | 근태 기록 보정 (HR 직권 보정) |
| `leave.approve` | 휴가 승인 |
| `leave.adjust` | 연차 수동 조정 |
| `policy.manage` | 근무/휴가 정책 관리 |
| `closing.execute` | 월 마감 실행/해제 |
| `report.dept / report.all` | 리포트 조회 범위 |

- 역할은 중복 부여 가능 (예: 팀장 = 일반 직원 + 승인자).
- 신규 요구 발생 시 역할 수정 없이 권한 추가/재배치로 대응 가능한 구조.
- **스코프 분리**: 데이터 접근 범위(self/dept/all)는 권한 문자열에 굳히지 않고 `권한(action) + 스코프(scope)` 두 필드로 분리해 저장한다. 1단계 스코프는 self/dept/all 3종이지만, 이 구조면 향후 "특정 부서 목록" 같은 세분화된 스코프를 권한 체계 재설계 없이 추가할 수 있다.

## 4. 기능 요구사항

### 4.1 직원/조직 관리

#### 4.1.1 직원 정보
| 항목 | 설명 |
|---|---|
| 사번 | 고유 식별자. ERP 병합 시 데이터 매핑 기준 |
| 기본 정보 | 이름, 이메일, 연락처, 입사일, 퇴사일, 재직 상태 |
| 직급 | 사원/대리/과장/차장/부장 등 — 코드 테이블로 관리 |
| 직책 | 팀장/본부장/실장 등 — 코드 테이블로 관리 (직급과 별개) |
| 고용형태 | 정규직 / 계약직 / 아르바이트 / 파견 등 |
| 근무형태 | 일반(사무실) / 재택 / 하이브리드 / 시차근무 등 |
| 근무정책 연결 | 개인별 근무정책 지정 (미지정 시 부서 → 전사 기본값 순으로 적용) |
| 연차정책 연결 | 개인별 연차정책 지정 (동일한 우선순위 적용) |
| 승인라인 연결 | 개인별 승인라인 지정 (미지정 시 부서 기본 승인라인) |

> **정책 적용 우선순위(공통 원칙)**: `직원 개인 지정 > 부서 지정 > 전사 기본값`. 대부분의 직원은 개인 지정 없이 부서/전사 기본값을 따르고, 예외 직원(임원, 특수 근무자)만 개인 지정한다.

#### 4.1.2 직원 변경 이력 (Employee History)
부서 이동, 직급/직책 변경, 고용형태/근무형태 변경은 **발효일 기준 이력**으로 관리한다.

- 변경 유형, 변경 전/후 값, 발효일, 사유를 `employee_histories`에 기록.
- 과거 시점의 근태·리포트는 **당시 소속/직급 기준으로 귀속** (예: 3월에 A팀이었던 직원의 3월 근태는 부서 이동 후에도 A팀 리포트에 집계).
- 감사 로그(누가 언제 바꿨나 — 시스템 관점)와 직원 이력(언제부터 무엇이 유효한가 — 업무 관점)은 목적이 다르므로 **둘 다 기록**한다.

#### 4.1.3 조직 관리
- 부서 트리 구조, 부서장 지정
- 부서 단위 기본 근무정책/승인라인 지정

#### 4.1.4 퇴사 프로세스
퇴사 처리는 퇴사일 입력으로 끝나지 않고, 아래 항목이 **하나의 프로세스**로 함께 처리된다.

| 순서 | 처리 | 설명 |
|---|---|---|
| 1 | 재직 상태 변경 | 재직 → 퇴사 예정 → 퇴사 (퇴사일 기준, 직원 이력 기록) |
| 2 | 로그인 비활성화 | 퇴사일 이후 로그인 차단 (계정·데이터는 보존) |
| 3 | 미래 근무일정 정리 | 퇴사일 이후 근무일정 삭제 |
| 4 | 미처리 신청 정리 | 본인이 신청한 승인 대기 건 자동 취소 처리 |
| 5 | 승인라인 정리 | 퇴사자가 승인자인 라인 목록 표시 → **대체 승인자 지정 완료 전 퇴사 처리 차단**, 퇴사자에게 대기 중이던 승인 건은 대체 승인자에게 재배정 |
| 6 | 연차 정산 | 잔여 연차 확정 및 정산 근거 데이터 제공 (미사용 수당 계산 자체는 급여 영역 — 제외 범위) |

- 퇴사자의 근태 기록·승인 이력은 삭제하지 않고 보존한다 (법정 보존 의무).
- 전 과정은 감사 로그 기록 대상.

### 4.3 근무일정 (Work Schedule) 관리

근태 판정(지각/조퇴/결근)의 **기준 데이터**. 모든 근태 계산은 근무일정과 실제 출퇴근 이벤트의 대조로 이루어진다.

| ID | 기능 | 설명 | 우선순위 |
|---|---|---|---|
| SCH-01 | 일정 자동 생성 | 근무정책 + 휴일 캘린더를 기반으로 직원×일자 단위 근무일정을 스케줄러가 배치 생성 (예: 매월 말 익월분) | 필수 |
| SCH-02 | 일정 개별 조정 | 특정 직원의 특정 일자 일정 변경 (휴무↔근무 전환, 시간 변경). 사유 필수, 이력 보존 | 필수 |
| SCH-03 | 일정 재생성 | 정책 변경·휴일 추가 시 미래 일정 재생성 (개별 조정분은 보존 여부 선택) | 필수 |
| SCH-04 | 시프트 지원 | 시프트(주간조/야간조 등) 정의 및 일정에 시프트 배정 — 데이터 구조만 우선 반영 | 선택 |

**근무일정이 답하는 질문**
- 오늘이 이 직원의 근무일인가, 휴무일인가?
- 근무일이라면 몇 시 출근 / 몇 시 퇴근 예정인가?
- → 이 기준이 있어야 지각/조퇴/결근을 자동 판정할 수 있다. 판정 규칙 상세는 5.2장(근태 계산 엔진).

### 4.4 출퇴근 관리

출퇴근은 하루 1쌍의 출/퇴근이 아니라 **이벤트 로그** 단위로 기록한다.

| ID | 기능 | 설명 | 우선순위 |
|---|---|---|---|
| ATT-01 | 체크 이벤트 기록 | 체크 유형: 출근 / 퇴근 / 외출 / 복귀 / 외근 시작·종료 / 출장 / 재택 시작·종료. 서버 시간 기준 저장 | 필수 |
| ATT-02 | 기록 메타데이터 | 이벤트마다 IP, 브라우저(User-Agent), 기기 정보 자동 수집. GPS는 정책 확정 시 선택 수집 | 필수 |
| ATT-03 | 기록 보정 신청 | 체크 누락·오류 시 사유(+첨부파일)와 함께 보정 신청 → 승인 후 반영. **원본 이벤트는 불변, 보정 이벤트를 별도 기록**하고 엔진이 보정본 우선 적용 | 필수 |
| ATT-04 | 오늘의 근태 현황 | 본인 이벤트 타임라인, 엔진 계산 근무시간, 주간 누적 근무시간 표시 | 필수 |
| ATT-05 | IP 대역 제한 | 사내 IP 대역에서만 체크 허용 (정책으로 on/off, 재택 근무형태는 예외 처리) | 선택 |

> **개인정보 유의사항**: IP·기기 정보·GPS는 개인정보에 해당하므로 수집 항목·목적·보존 기간을 사내 고지하고 동의 절차를 거친다. GPS는 필요성이 확정되기 전까지 수집하지 않는다(기본 off).

### 4.5 연차/휴가 관리

#### 4.5.1 휴가 유형 (leave_types — 정책 데이터)
| 유형 | 단위 | 유급 여부 | 첨부파일 | 비고 |
|---|---|---|---|---|
| 연차 | 1일 / 0.5일(반차) | 유급 | — | 연차정책 기준 자동 발생 |
| 대체휴가 | 1일 / 0.5일 | 유급 | — | 휴일근무 승인 건에서 발생 |
| 경조휴가 | 일 단위 | 유급 | 선택 | 유형별 일수 정책 설정 |
| 병가 | 일 단위 | 정책 설정 | **필수 옵션** (진단서) | 첨부 필수 여부는 유형별 설정 |
| 공가 | 일 단위 | 유급 | 선택 (예비군 통지서 등) | |
| 무급휴가 | 일 단위 | 무급 | — | |

> 휴가 유형은 HR이 관리 화면에서 추가/수정하는 정책 데이터. 유형별로 `첨부파일 필수 여부`, `증빙 요구 조건`을 설정할 수 있다.

#### 4.5.2 연차정책 (leave_policies — 유형과 분리된 정책 테이블)
연차의 **발생·소멸 규칙**은 휴가 유형과 분리하여 정책으로 관리한다. 직원별/전사 기본값으로 배정 가능 (4.1.1 우선순위 규칙).

| 정책 항목 | 설정 값 예시 |
|---|---|
| 부여 기준 | **입사일 기준 (확정 — 전사 기본값)**. 회계연도 기준도 구조상 지원하되 1단계 UI 미개발 |
| 발생 규칙 | 근로기준법 제60조 기본 (1년 미만 월 1일 최대 11일 / 1년 이상 15일 / 2년마다 +1일 최대 25일) |
| 사용기한 | 발생일로부터 1년 (설정 가능) |
| 이월 | 허용 여부, 이월 한도 |
| 소멸 | 기한 도래 시 자동 소멸 (소멸 내역 기록) |
| 촉진 | 촉진 알림 시점 (D-60/D-30 등), 대상 조건 |
| 사용 단위 | 1일 / 반차(0.5) / 반반차(0.25, 허용 시) |

- 고용형태별로 다른 정책 적용 가능 (예: 아르바이트는 별도 정책).
- 정책 변경은 적용 시작일 기준 버전 관리 — 기발생 연차에는 소급 적용하지 않는다.

#### 4.5.3 휴가 신청/승인
| ID | 기능 | 설명 | 우선순위 |
|---|---|---|---|
| LEV-01 | 휴가 신청 | 유형, 기간, 반차 여부, 사유, 첨부파일(유형별 필수 설정) 입력. 잔여일수 초과·기간 중복 신청 시 차단 | 필수 |
| LEV-02 | 승인/반려 | 승인라인에 따른 다단계 승인. 반려 시 사유 필수. 결과 알림 발송 | 필수 |
| LEV-03 | 신청 취소 | 승인 전 자유 취소, 승인 후 시작일 전이면 취소 신청(승인 필요). 상태 전이는 5.3장 | 필수 |
| LEV-04 | 잔여 연차 조회 | 발생 건별 발생/사용/잔여/소멸 예정 내역 조회 | 필수 |
| LEV-05 | 부서 휴가 캘린더 | 부서원 휴가 일정 월 캘린더 표시 (승인자 판단 지원) | 필수 |
| LEV-06 | 연차 수동 조정 | HR이 발생/차감 수동 조정 (사유 필수, 감사 로그 기록) | 필수 |

- 휴가 일수 계산 시 **근무일정 기준으로 휴무일(주말·휴일) 자동 제외**.
- 잔여 차감은 승인 시점 확정, 신청 시점에는 가차감(pending)으로 이중 신청 방지.
- 승인된 휴가는 엔진 입력으로 전달되어 해당 일자의 근태 판정에 자동 반영 (결근 아님).

### 4.7 승인라인 관리

처음부터 **승인라인 템플릿 + 단계** 구조로 설계한다.

| ID | 기능 | 설명 | 우선순위 |
|---|---|---|---|
| APV-01 | 승인라인 정의 | 승인라인(템플릿) 등록: 이름, 적용 대상(신청 유형), 단계 목록 | 필수 |
| APV-02 | 단계 정의 | 각 단계의 승인자 지정 방식: 특정인 / 직속 부서장 / 상위 부서장 / 특정 직책 | 필수 |
| APV-03 | 라인 배정 | 직원별/부서별 승인라인 배정 (4.1.1 우선순위 규칙) | 필수 |
| APV-04 | 승인 처리 | 순차 승인, 중간 반려 시 종료. 승인자 부재 시 대결(위임) 설정 | 필수 (대결은 선택) |
| APV-05 | 승인함 | 승인자별 대기 건 목록, 일괄 승인 | 필수 |

**예시**: `일반 휴가 라인` = 팀장(1단계) / `장기 휴가 라인` = 팀장 → 본부장 / `임원 라인` = 대표

- 승인 구조는 휴가·초과근무·근태 보정이 공통으로 사용하는 **독립 모듈**로 구현한다 (10장).

### 4.8 첨부파일
| ID | 기능 | 설명 | 우선순위 |

| 대상 | Import | Export | 비고 |
|---|---|---|---|
| 직원 | ○ | ○ | 사번 기준 upsert |
| 부서 | ○ | ○ | |
| 연차 초기값 | ○ | ○ | 오픈 시점 잔여 연차를 `leave_grants` 초기 발생 건으로 등록 (미확정 사항 #9) |
| 휴일 | ○ | ○ | 연도별 |
| 승인라인 | ○ | ○ | |

- Import는 **2단계 처리**: 업로드 → 검증 결과 미리보기(오류 행·사유 표시) → 확정 반영. 오류 행만 수정 후 재업로드 가능.

- **확장 대비**: 내부 구조를 `파싱(포맷별) → 검증(공통) → 반영(공통)` 3단계로 분리한다. 1단계는 Excel 파서만 구현하되, 이 구조면 향후 CSV·API·배치 연동(Data Integration)이 파서/입력 어댑터 추가만으로 가능하다. 범용 연동 플랫폼을 지금 만들지는 않는다.

## 5. 아키텍처 설계

### 5.1 ERP 공통 설계 원칙 — Master / Transaction / Engine / Snapshot

근태 모듈에서 정립하는 아래 패턴을 향후 ERP 전 모듈의 공통 설계 원칙으로 삼는다.

| 레이어 | 정의 | 규칙 |
|---|---|---|
| **Master** (기준정보) | 정책·기준이 되는 데이터 | 변경 시 적용 시작일 기준 버전 관리, 소급 적용 금지 |
| **Transaction** (트랜잭션) | 실제 발생한 사실의 기록 | 불변(append-only), 수정 대신 보정/취소 레코드 추가 |
| **Engine** (계산) | Master × Transaction → 결과 | 순수 계산, DB 쓰기 없음, 파생값 저장 금지 |
| **Snapshot** (마감) | 기간 확정 시점의 계산 결과 | 마감 후 불변, 후속 프로세스(급여 등)의 근거 |

**모듈별 적용 예시** — 동일한 설계 철학이 회계·인사 등 향후 모듈에도 적용된다.
| 모듈 | Master | Transaction | Engine | Snapshot |
|---|---|---|---|---|
| 근태 | 근무정책, 근무일정 | 출퇴근 이벤트 | 근태 계산 엔진 | 월마감 스냅샷 |
| 휴가 | 연차정책, 휴가유형 | 발생(grants)/사용(usages) | 잔여 계산 | 마감 시 휴가 집계 |
| 회계 (향후) | 계정과목 (Accounts) | 분개 (Journal Entries) | 시산/집계 | 기간 마감 (Period Closing) |
| 인사 (향후) | 직급/보상 정책 | 인사 발령 이력 | 평가/보상 계산 | 확정 인사 데이터 |

**공통 보조 원칙**
- 모든 Master 변경·Transaction 보정은 감사 로그(before/after) 기록
- 시간 흐름이 있는 속성(소속, 직급, 정책)은 발효일 기준 이력 관리
- **정책 공통 규약**: 모든 정책류 Master(근무정책, 연차정책, 향후 급여정책·평가정책 등)는 ① 적용 시작일 기준 버전 관리, ② 배정 우선순위(개인 > 부서 > 전사 기본값), ③ 변경 감사 로그 — 이 세 가지 규약을 공통으로 따른다. 단, 정책들의 **구조(스키마)는 정책마다 다르므로 상위 공통 테이블로 묶지 않는다** — 공유하는 것은 규약(패턴)이지 데이터 구조가 아니다
- **공통 코드(Code Master) 원칙**: 단순 열거값(근무형태, 파일 유형, 알림 유형, 부서 유형 등)은 개별 테이블을 만들지 않고 공통 코드 테이블(`common_codes`: 코드 그룹 + 코드)로 관리한다. 단, **구조적 속성을 가진 데이터는 공통 코드가 아니라 전용 정책 테이블**로 관리한다 (예: 휴가 유형은 유급 여부·첨부 필수 등의 속성이 있으므로 `leave_types` 유지). "모든 선택값의 코드화"가 아니라 "속성 없는 열거값의 코드화"가 기준이다

> 이 원칙은 ERP 병합 시점에 별도 문서(`docs/architecture/erp-design-principles.md`)로 추출하여 ERP 표준 아키텍처 문서로 승격한다.

### 5.2 근태 계산 엔진 (Attendance Engine)

근태 판정 로직을 서비스 곳곳에 흩어놓지 않고 **하나의 독립 계산 모듈**로 구현한다.

**입력 / 출력**
```
입력:  대상 직원 × 기간
       + 해당 기간의 근무정책 버전
       + 근무일정
       + 출퇴근 이벤트 (보정 반영본)
       + 승인된 휴가 / 초과근무 / 출장·외근

출력:  일별 근태 결과 (판정: 정상/지각/조퇴/결근/휴가/휴무,
       근무시간, 지각시간, 초과근무시간)
       + 기간 집계 (주간/월간 합계)
```

**엔진 계약(Contract)**
1. **순수 계산**: DB에 쓰지 않는다. 같은 입력이면 항상 같은 출력 (결정적).
2. **단일 판정 지점**: 화면 조회, 52시간 모니터링, 마감 스냅샷 생성이 모두 같은 엔진을 호출한다. 판정 로직이 두 곳에 존재하지 않는다.
3. **정책 버전 인지**: 계산 대상 일자에 유효했던 정책 버전을 사용한다 (소급 계산 정확성).
4. **테스트 우선**: 판정 규칙별 단위 테스트를 케이스 표로 관리 (지각/조퇴/결근/자율출퇴근/휴가 겹침/자정 넘김 근무 등). 연차 발생 계산도 동일 원칙의 별도 엔진(leave engine)으로 구현.

**판정 규칙 (근무일정 × 이벤트 대조)**
| 판정 | 조건 |
|---|---|
| 정상 | 예정 출근시각 이내 체크인 + 예정 퇴근시각 이후 체크아웃 |
| 지각 | 예정 출근시각 + 유예시간(정책) 이후 체크인 |
| 조퇴 | 예정 퇴근시각 이전 체크아웃 (승인된 조퇴/반차 제외) |
| 결근 | 근무일인데 출퇴근 이벤트 없음 (승인된 휴가/출장 제외) |
| 휴가/휴무 | 승인된 휴가일 또는 일정상 휴무일 — 판정 제외 |
| 자율출퇴근 | 지각/조퇴 판정 없음, 근무시간만 집계 |

> **참고**: 별도의 범용 Rule Engine 레이어는 두지 않는다. 판정 규칙은 근무정책(Master)의 설정값으로 파라미터화하고 엔진 코드가 해석한다. 규칙 자체를 데이터로 정의하는 범용 룰엔진은 현재 요구 수준에서 과설계이며, 정책 유형 추가로 대응 불가한 요구가 확인되면 그때 도입을 검토한다.

### 5.3 상태(State) 정의

주요 업무 객체의 상태와 전이를 정의한다. **정의된 전이 외의 상태 변경은 시스템이 거부**하며, 모든 전이는 이력(처리자, 일시)을 남긴다.

#### 휴가 신청 / 초과근무 신청
```
REQUESTED ──승인(전 단계 완료)──▶ APPROVED
    │ ├──반려──▶ REJECTED
    │ └──신청자 취소──▶ CANCELLED
    │
APPROVED ──시작일 전 취소 신청──▶ CANCEL_REQUESTED ──승인──▶ CANCELLED (가차감 복원)
                                        └──반려──▶ APPROVED (유지)
```
| 상태 | 설명 | 전이 주체 |
|---|---|---|
| REQUESTED | 신청 제출, 승인 대기 (잔여 가차감) | 신청자 |
| APPROVED | 전 승인 단계 완료 (차감 확정) | 최종 승인자 |
| REJECTED | 반려 (가차감 복원) | 승인자 |
| CANCELLED | 취소 (차감 복원) | 신청자 / 취소 승인자 |
| CANCEL_REQUESTED | 승인 후 취소 요청 중 | 신청자 |

> 임시저장(DRAFT) 상태는 두지 않는다 — 신청 즉시 제출 흐름으로 충분하며, 임시저장 요구가 확인되면 추가한다.

#### 근태 보정 신청
```
REQUESTED ──승인──▶ APPROVED ──보정 이벤트 생성──▶ APPLIED
    ├──반려──▶ REJECTED
    └──신청자 취소──▶ CANCELLED
```
- `APPROVED`와 `APPLIED`를 분리하는 이유: 승인(사람의 판단)과 반영(시스템의 이벤트 생성)을 구분하여, 반영 실패 시 재처리 가능하게 한다.

#### 승인 인스턴스 (approval)
```
IN_PROGRESS ──단계 승인──▶ IN_PROGRESS (다음 단계) ──최종 단계 승인──▶ APPROVED
    ├──반려──▶ REJECTED (즉시 종료)
    └──원 신청 취소──▶ CANCELLED
```

#### 월 마감 (monthly_closing)
```
OPEN ──마감 실행──▶ VALIDATING ──검증 통과 + 스냅샷 생성──▶ CLOSED
                        └──검증 실패──▶ OPEN (이상 건 목록 제공)

CLOSED ──HR 마감 해제(사유 필수)──▶ REOPENED ──재마감──▶ VALIDATING → CLOSED (스냅샷 재생성)
```
| 상태 | 설명 |
|---|---|
| OPEN | 해당 월 기록 입력/수정 가능 |
| VALIDATING | 자동 검증 실행 중 |
| CLOSED | 확정 — 해당 월 기록 수정 불가, 스냅샷 유효 |
| REOPENED | 마감 해제 — 수정 가능, 기존 스냅샷 무효 표시 |

#### 연차 발생 건 (leave_grant)
```
ACTIVE ──전량 사용──▶ EXHAUSTED
   └──사용기한 도래──▶ EXPIRED (잔여분 소멸 기록)
```

### 5.4 공통 스케줄러 (Scheduler) — ERP Core Service

배치 작업을 기능별로 흩어놓지 않고 **공통 스케줄러 모듈**이 등록·실행·이력 관리를 담당한다.

> **위상**: 스케줄러는 근태 모듈의 부속이 아니라 **ERP 전체의 핵심 공통 서비스(Core Service)**다. 근태에서 먼저 구축·검증하지만, 향후 회계 자동 마감, 자동 분개, 미수금 점검, 정산 작업, 보고서 생성 등 ERP 전 모듈의 배치가 동일 모듈 위에서 실행된다. 따라서 스케줄러 코드에는 근태 도메인 지식을 넣지 않는다 (작업 등록 인터페이스만 제공, 작업 내용은 각 모듈이 구현).

**등록 작업 (1단계)**
| 작업 | 주기 (기본값) | 내용 |
|---|---|---|
| 근무일정 생성 | 매월 말 | 익월분 직원×일자 일정 생성 |
| 연차 발생 | 매일 | 입사일/월 개근/연 기준 도래 직원 연차 부여 |
| 연차 소멸 | 매일 | 사용기한 도래 grant를 EXPIRED 처리, 소멸 기록 |
| 연차 촉진 알림 | 매일 | 정책상 촉진 시점 도래 대상자 알림 발행 |
| 주 52시간 체크 | 매일 | 주간 누적 근무시간 엔진 계산 → 임계 초과자 알림 발행 |
| 알림 발송 처리 | 상시 (짧은 주기) | 아웃박스 큐의 대기 알림 발송 (5.5장) |

**공통 요건**
- 작업 정의(`scheduler_jobs`)와 실행 이력(`scheduler_job_runs`: 시작/종료, 성공/실패, 처리 건수, 오류)을 기록
- 실패 시 재시도 정책, 관리자 수동 실행 지원
- **멱등성**: 같은 작업이 같은 날 두 번 실행되어도 중복 발생/중복 알림이 없어야 한다 (예: 연차 발생은 발생 기준일+직원 유니크 제약)

### 5.5 알림 아웃박스 큐 (Notification Outbox) — ERP Core Service

알림의 **발행(무엇을 알릴 것인가)**과 **발송(어떤 채널로 보낼 것인가)**을 분리한다.

> **위상**: 알림도 스케줄러와 마찬가지로 **ERP 공통 서비스**다. 향후 회계 승인, 구매 승인, 전자결재, 재고 부족, 미수금 알림 등이 모두 같은 아웃박스·채널 어댑터를 사용한다. 알림 모듈은 "무엇을 알리는가"를 알지 못하며, 템플릿과 수신자만 받아 발송한다.

```
업무 이벤트 발생 (승인 요청, 촉진 대상 등)
      │  발행 — 업무 트랜잭션과 동일 트랜잭션에 기록
      ▼
notification_outbox (수신자, 템플릿, 파라미터, 채널, 상태: PENDING)
      │  발송 — 스케줄러가 큐 폴링
      ▼
채널 어댑터 (1단계: 시스템 내 알림, 이메일)
      │
      ▼
상태 갱신 (SENT / FAILED → 재시도, 최대 횟수 초과 시 DEAD)
```

- 업무 트랜잭션과 알림 발행이 원자적 → 알림 유실 없음. 발송 실패는 재시도로 복구 → 업무 처리가 이메일 서버 장애에 영향받지 않음.
- 채널 어댑터 인터페이스로 Slack/카카오워크/SMS 확장 — 어댑터 1개 추가로 대응, 업무 코드 무변경.
- 1단계는 **DB 테이블 기반 큐**로 충분하다. 메시지 브로커(Kafka/RabbitMQ 등) 도입은 현 규모에서 과설계이며, ERP 병합 후 전사 알림량 기준으로 재검토한다.

## 6. 비기능 요구사항

| 항목 | 요구사항 |
|---|---|
| 보안 | 비밀번호 해시 저장(bcrypt 등), 기능 단위 권한 기반 RBAC, 근태 기록 접근은 권한자만 |
| 개인정보 | IP·기기 정보 등 수집 항목·목적·보존 기간 사내 고지 및 동의. GPS 기본 미수집. 첨부파일(진단서 등 민감 문서) 접근 통제 |
| 감사 이력 | 주요 변경(보정, 연차 조정, 일정 조정, 마감/해제, 정책 변경, 시스템 설정, Import)은 **변경 전/후 값(before/after), 변경 사유, 변경자, 변경 일시, 요청 IP**를 감사 로그로 보존. **변경 사유는 필수 입력** — 사유 없는 직권 변경은 시스템이 거부한다 |
| 접속 기록 | 변경 로그와 별도로 **접속 로그(access_logs)**를 기록: 로그인/로그아웃 이력, 개인정보(타인 근태·직원 정보) 조회 이력. 개인정보보호법령상 개인정보처리시스템 접속기록 보관 의무(최소 1년) 준수. 성격·볼륨이 다르므로 감사 로그와 테이블 분리 |
| 데이터 보존 | 근태 기록 3년 이상 보존 (근로기준법 관련 서류 보존 의무 고려) |
| 정합성 | 파생값은 저장하지 않고 엔진 계산 (5.2장). 단 월마감 스냅샷은 예외 (1.4장). 상태 전이는 정의된 경로만 허용 (5.3장) |
| 시간 기준 | 서버 시간(KST) 단일 기준. 클라이언트 시간 미신뢰 |
| 가용성 | 출퇴근 체크가 몰리는 시간대(08~10시, 17~19시) 응답 지연 없도록 |
| 사용성 | 반응형 웹 (모바일 브라우저에서 출퇴근 체크·휴가 신청 가능) |

```
access_logs          접속 로그 (로그인/로그아웃, 개인정보 조회 이력 — 변경 로그와 분리,
                      법정 보관 기간 준수)
monthly_closings     월 마감 (대상 월, 상태: 5.3장, 실행자, 검증 결과)
closing_snapshots    마감 스냅샷 (직원×월: 근무시간, 지각/조퇴/결근 횟수,           [Snapshot]
                      초과근무 시간, 휴가 사용 집계 — 마감 후 불변, 급여 근거)
```

**설계 원칙 요약** (5.1장 공통 원칙의 적용)
- 파생값 미저장: 잔여 연차 = `leave_grants − leave_usages` 계산, 근태 판정 = 엔진 계산. 성능 이슈 시 무효화 가능한 캐시 도입, 원본 우선.
- 마감 스냅샷 예외: 급여 근거는 마감 시점 값으로 고정.
- 원본 불변: 출퇴근 이벤트·승인 이력은 수정하지 않고 보정/취소 레코드를 추가.
- 상태 전이 강제: 5.3장에 정의된 전이만 허용.
- 근태 도메인 테이블은 `users`가 아닌 `employees`(사번)를 참조 → ERP 병합 시 계정 체계 변경에 영향 없음.
- **단일 법인 전제 (1.3장)**: 어떤 테이블에도 `company_id`를 두지 않는다. 법인 고유 값은 `system_settings`로 분리한다.

## 10. 모듈 구조 및 ERP 병합 전략

### 10.1 daolERP 전체 구조 (장기 로드맵)

근태 시스템은 daolERP의 첫 모듈이며, 여기서 구축하는 공통 서비스(Core)가 이후 전 모듈의 기반이 된다.

```
┌─────────────────────── ERP Core (공통 서비스) ───────────────────────┐
│  User / Employee / Department   Approval (승인)    Notification (알림)  │
│  Permission (RBAC)              Scheduler (배치)    Attachment (파일)   │
│  Code Master (공통 코드)         Audit / Access Log  System Settings    │
└──────────────────────────────┬──────────────────────────────────────┘
                               │ 전 모듈이 공통 사용
   ┌───────────┬───────────────┼───────────────┬─────────────┬──────────┐
   ▼           ▼               ▼               ▼             ▼          ▼
 Attendance   Leave          Payroll        Accounting    Inventory   Sales /
 (근태)       (휴가)          (급여)          (회계)         (재고)      Purchase
 [본 시스템]   [본 시스템]     (향후)          (향후)         (향후)      (향후)

 구축 순서: Attendance + Leave (본 시스템) → Payroll → Accounting → Inventory → Sales/Purchase
 설계 원칙: 전 모듈이 Master → Transaction → Engine → Snapshot 패턴(5.1장)을 따른다
 법인 전개: 완성된 ERP를 법인별로 복제하여 독립 운영 (1.3장 — 멀티테넌트 아님)
```

- 본 시스템(근태+휴가)에서 ERP Core의 실구현체가 처음 만들어진다 — 승인, 알림, 스케줄러, 첨부, 권한, 감사 로그가 여기서 검증된 뒤 ERP 공통 서비스로 승격된다 (10.3).
- 각 업무 모듈은 Core에만 의존하고 서로 직접 의존하지 않는 것을 원칙으로 한다 (모듈 간 데이터는 마감 스냅샷 등 확정 데이터로 전달 — 예: 급여는 근태 마감 스냅샷을 입력으로 사용).

### 10.2 근태 시스템 모듈 구조
병합 시 영향도를 최소화하기 위해 도메인별 독립 모듈로 구성한다. 모듈 간 참조는 인터페이스를 통해서만 한다.

```
common             사용자/직원/조직/권한  ← 2단계에서 ERP 공통 모듈로 교체되는 유일한 부분
policy             근무정책/연차정책/휴가유형/휴일
schedule           근무일정/시프트
attendance         출퇴근 이벤트/보정
attendance-engine  근태 계산 엔진 (순수 계산 — attendance와 분리하여 독립 테스트)
leave              연차 발생/신청/잔여 (+ leave engine)
overtime           초과근무
approval           승인라인/승인 처리      ← ERP 타 모듈(전자결재 등)에서 재사용 가능
attachment         첨부파일               ← 범용 구조, ERP 전반 재사용 가능
scheduler          공통 스케줄러           ← ERP 전 모듈의 배치 실행 기반
notification       알림 (아웃박스 큐)      ← 2단계에서 ERP 공통 알림으로 통합
closing            월 마감/스냅샷
report             통계/리포트
```

### 10.3 병합 절차 (2단계)
- ERP 공통 모듈(사용자/조직/권한)이 준비되면 `common` 모듈 구현체만 ERP 것으로 교체
- 직원/부서 데이터 매핑 마이그레이션 (사번 기준)
- 나머지 모듈의 테이블은 데이터 이관 없이 스키마째 이동 (동일 DB라면 그대로 사용)
- `approval`/`attachment`/`scheduler`/`notification`은 ERP 공통 서비스로 승격 후보 — 병합 시점에 ERP 표준으로 채택 여부 결정. 승격 시 `attachment`는 계약서·세금계산서·품의서·인사서류 등 전 모듈의 파일을 담당하는 **File Service**로 명칭·역할 확장 (현재의 참조 유형 + 참조 ID 범용 구조가 그대로 수용 가능)
- 5.1장 설계 원칙을 ERP 표준 아키텍처 문서로 추출
- 화면은 ERP 셸(공통 레이아웃/메뉴) 안에 근태 메뉴로 편입

### 10.4 검증 기준 (병합 GO/NO-GO)
독립 운영 기간 중 아래 충족 시 병합 진행:
- 1회 이상 월 마감을 실데이터로 무리 없이 완료 (자동 검증 → 마감 → 스냅샷 → 내보내기 전체 사이클)
- 연차 자동 발생 결과가 수기 계산과 전수 일치
- 엔진의 근태 판정(지각/조퇴/결근)이 실제와 일치하는지 표본 대조 완료
- 사용자 불편 사항(버그/개선) 처리 완료

## 11. 개발 단계 및 일정 (안)

기준 데이터가 먼저 만들어지는 순서로 진행한다: **조직 → 정책 → 일정 → 승인 골격 → 출퇴근/엔진 → 휴가 → 리포트/마감**.

| 단계 | 기간(안) | 내용 |
|---|---|---|
| M1 | 2주 | 프로젝트 셋업, 공통 도메인(사용자/직원(확장 필드+이력)/조직/직급·직책/권한/로그인/접속 로그/공통 코드), **시스템 설정**, 화면 골격, **공통 스케줄러 골격** |
| M2 | 2주 | 근무정책·연차정책·휴가유형·휴일 관리, 정책 배정(직원/부서/기본값), 정책 버전 관리 |
| M3 | 2주 | 근무일정: 자동 생성 배치, 개별 조정, 재생성 + **승인 모듈 골격**(라인/단계/승인함/상태기계) |
| M4 | 3주 | 출퇴근: 이벤트 기록(메타데이터 포함), 보정 신청·승인, **근태 계산 엔진**(판정 규칙 + 단위 테스트 케이스 표), 첨부파일 |
| M5 | 3주 | 휴가: 연차 발생/소멸 엔진·배치, 신청·승인, 잔여 관리, 부서 캘린더, 촉진 알림 |
| M6 | 3주 | 초과근무, 52시간 모니터링, 알림 아웃박스 큐 + 채널 어댑터(시스템 내/이메일), **퇴사 프로세스**(연차 정산 포함) |
| M7 | 3주 | 월 마감(자동 검증 + 상태기계 + 스냅샷 + 내보내기), 통계/리포트, **관리자 대시보드**, **Excel Import/Export** |
| M8 | 1주 | 통합 테스트, 실데이터 검증(연차 계산 전수 대조, 엔진 판정 표본 대조), **초기 데이터 Import 리허설**, 배포 |
| 운영 검증 | 1~3개월 | 실사용 및 피드백 반영 → 병합 GO/NO-GO 판단 |

> 총 개발 약 19주 + 운영 검증 (v0.3의 17주 대비 +2주 — 퇴사 프로세스, 관리자 대시보드, Excel Import/Export 반영). 인력 규모에 따라 조정.
