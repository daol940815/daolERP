## 1. 개요

### 1.1 배경 및 목적
- 출퇴근 기록, 연차/휴가 관리, 초과근무 관리 등 근태 업무를 수기(엑셀, 메신저 결재 등)로 처리하면서 발생하는 누락·오류·이력 관리의 어려움을 해소한다.
- 우선 **독립 실행 가능한 근태 관리 시스템**으로 개발·운영하고, 실사용 검증 후 문제가 없으면 **daolERP 본체에 모듈로 병합**한다.
- 연차 발생/소진 계산을 자동화하여 근로기준법 준수와 인사 담당자의 업무 부담 경감을 동시에 달성한다.

### 1.2 개발 전략 (2단계)
| 단계 | 내용 |
|---|---|
| 1단계 (독립 운영) | 근태 관리 단독 시스템으로 개발. 자체 로그인/사용자 관리 포함. 실무에서 사용하며 검증 |
| 2단계 (ERP 병합) | 검증 완료 후 daolERP의 한 모듈로 병합. 사용자/조직/권한은 ERP 공통 모듈로 이관 |

> **병합을 전제로 한 설계 원칙**: 사용자·조직·권한 등 공통 도메인은 근태 도메인과 코드/DB 레벨에서 분리해 두어, 병합 시 근태 도메인 코드는 최소 수정으로 이식 가능하도록 한다. (상세: 9장)

### 1.3 핵심 설계 구조

시스템의 근간이 되는 계산 구조는 다음과 같다.

```
근무정책 (work_policy)          "이 직원은 어떤 규칙으로 일하는가"
      │  배치 생성 + 개별 조정
      ▼
근무일정 (work_schedule)        "이 직원은 오늘 몇 시부터 몇 시까지 근무 예정인가"
      │  대조
      ▼
출퇴근 이벤트 (attendance_event) "실제로 언제 출근/퇴근/외출했는가" (Raw Data)
      │  계산 (저장하지 않음)
      ▼
근태 판정                        지각/조퇴/결근/근무시간/초과근무시간
      │  월마감 시점에 확정
      ▼
마감 스냅샷 (closing_snapshot)   급여 계산의 근거 (마감 후 불변)
```

- **원본(Raw) 데이터만 저장하고 파생값(근무시간, 지각시간 등)은 저장하지 않는다.** 정책 변경 시 재계산이 가능하고 정합성이 유지된다.
- **단, 월마감 시점에는 계산 결과를 스냅샷으로 확정 저장한다.** 급여 계산의 근거는 이후 정책이 바뀌어도 변하지 않아야 하기 때문이다. (마감 전 = 항상 계산, 마감 후 = 스냅샷 조회)

### 1.4 용어 정의
| 용어 | 정의 |
|---|---|
| 근태 | 출근, 퇴근, 지각, 조퇴, 결근, 외근, 재택 등 근무 상태 전반 |
| 근무정책 | 근무 형태(고정/시차/자율), 기준 출퇴근 시각, 휴게시간, 지각 판정 기준 등의 규칙 묶음 |
| 근무일정 | 직원×일자 단위로 생성되는 근무 예정 정보 (근무일 여부, 예정 출퇴근 시각) |
| 시프트 | 교대근무 등에서 사용하는 근무 시간대 단위 (주간조/야간조 등) |
| 연차 | 근로기준법상 연차유급휴가 |
| 반차 | 연차 0.5일 단위 사용 (오전/오후) |
| 대체휴가 | 휴일근무에 대한 보상 휴가 |
| 근태 마감 | 월 단위로 근태 기록을 확정하여 급여 계산 등에 사용하는 것 |

## 2. 범위

### 2.1 포함 범위 (1단계)
1. 직원/조직 관리 (직급, 직책, 고용형태, 정책 연결 포함)
2. 근무정책 관리 (직원별/부서별 상이한 정책 적용)
3. 근무일정 관리 (정책 기반 자동 생성 + 개별 조정)
4. 출퇴근 기록 (이벤트 단위 기록, 수동 보정 신청)
5. 연차/휴가 관리 (정책 기반 자동 발생, 신청, 승인, 잔여 관리, 촉진 알림)
6. 초과근무/휴일근무 신청 및 승인
7. 승인라인 관리 (다단계 결재)
8. 첨부파일 (병가 진단서, 출장계획서 등)
9. 근태 현황 조회 및 월 마감 (자동 검증 포함)
10. 통계/리포트 (개인별·부서별·기간별, HR 활용 리포트)
11. 관리자 기능 (사용자/조직/정책/휴일 관리)
12. 알림 (승인 요청/결과, 연차 촉진 등)
13. 감사 로그 (before/after 포함)

### 2.2 제외 범위 (추후 검토)
- 급여 계산 연동 (마감 스냅샷 내보내기까지만 제공)
- 출입 통제 장비(카드/지문) 연동 — 2단계 이후 검토
- 모바일 네이티브 앱 (반응형 웹으로 대체)
- GPS 기반 위치 기록 — 개인정보 이슈 검토 후 결정 (11장 참조)
- 교대근무(시프트) 자동 편성 — 시프트 데이터 구조는 반영하되, 편성 자동화는 실수요 확인 후

## 3. 사용자 및 권한

### 3.1 역할
| 역할 | 설명 |
|---|---|
| 일반 직원 | 전 임직원. 본인 출퇴근 기록, 휴가/초과근무 신청, 본인 현황 조회 |
| 승인자 | 승인라인 상의 결재 권한자 (팀장/본부장/대표 등). 담당 신청 건 승인/반려, 소속 조직 현황 조회 |
| 인사 담당자 (HR) | 전사 근태 조회, 근태 보정, 정책 설정, 월 마감, 리포트 |
| 시스템 관리자 | 사용자/조직/권한 관리, 시스템 설정 |

### 3.2 권한 설계 — 기능 단위 권한 기반 RBAC
메뉴 접근 권한만으로는 부족하므로, **기능 단위 권한(permission)을 정의하고 역할(role)은 권한의 묶음**으로 설계한다.

| 권한 예시 | 설명 |
|---|---|
| `attendance.read.self / dept / all` | 근태 조회 범위 |
| `attendance.correct` | 근태 기록 보정 (HR 직권 보정) |
| `leave.approve` | 휴가 승인 |
| `leave.adjust` | 연차 수동 조정 |
| `policy.manage` | 근무/휴가 정책 관리 |
| `closing.execute` | 월 마감 실행/해제 |
| `report.dept / report.all` | 리포트 조회 범위 |

- 역할은 중복 부여 가능 (예: 팀장 = 일반 직원 + 승인자).
- 신규 요구 발생 시 역할 수정 없이 권한 추가/재배치로 대응 가능한 구조.

## 4. 기능 요구사항

### 4.1 직원/조직 관리

#### 4.1.1 직원 정보
| 항목 | 설명 |
|---|---|
| 사번 | 고유 식별자. ERP 병합 시 데이터 매핑 기준 |
| 기본 정보 | 이름, 이메일, 연락처, 입사일, 퇴사일, 재직 상태 |
| 직급 | 사원/대리/과장/차장/부장 등 — 코드 테이블로 관리 |
| 직책 | 팀장/본부장/실장 등 — 코드 테이블로 관리 (직급과 별개) |
| 고용형태 | 정규직 / 계약직 / 아르바이트 / 파견 등 |
| 근무형태 | 일반(사무실) / 재택 / 하이브리드 / 시차근무 등 |
| 근무정책 연결 | 개인별 근무정책 지정 (미지정 시 부서 → 전사 기본값 순으로 적용) |
| 연차정책 연결 | 개인별 연차정책 지정 (동일한 우선순위 적용) |
| 승인라인 연결 | 개인별 승인라인 지정 (미지정 시 부서 기본 승인라인) |

> **정책 적용 우선순위(공통 원칙)**: `직원 개인 지정 > 부서 지정 > 전사 기본값`. 대부분의 직원은 개인 지정 없이 부서/전사 기본값을 따르고, 예외 직원(임원, 특수 근무자)만 개인 지정한다.

#### 4.1.2 조직 관리
- 부서 트리 구조, 부서장 지정
- 부서 단위 기본 근무정책/승인라인 지정
- 직원 부서 이동 이력 관리 (근태 데이터는 이동 시점 기준으로 부서 귀속)

### 4.2 근무정책 관리

| ID | 기능 | 설명 | 우선순위 |
|---|---|---|---|
| POL-01 | 근무정책 등록 | 정책 유형(고정 출퇴근 / 시차출퇴근 / 자율출퇴근), 기준 출퇴근 시각, 휴게시간, 소정근로시간, 지각/조퇴 판정 기준 | 필수 |
| POL-02 | 정책 배정 | 직원별/부서별 정책 지정 (4.1.1 우선순위 규칙) | 필수 |
| POL-03 | 정책 변경 이력 | 정책 내용 변경 시 적용 시작일 기준으로 버전 관리 — 과거 근태는 당시 정책으로 계산 | 필수 |

**정책 예시**
| 정책명 | 유형 | 근무시간 | 대상 |
|---|---|---|---|
| 사무직 표준 | 고정 | 09:00~18:00 (휴게 1h) | 전사 기본값 |
| 생산직 표준 | 고정 | 08:30~17:30 (휴게 1h) | 생산부서 |
| 시차출퇴근 A | 시차 | 08~10시 사이 출근, 8h 근무 | 개인 지정 |
| 임원 | 자율 | 코어타임 없음, 지각 판정 없음 | 개인 지정 |

### 4.3 근무일정 (Work Schedule) 관리

근태 판정(지각/조퇴/결근)의 **기준 데이터**. 모든 근태 계산은 근무일정과 실제 출퇴근 이벤트의 대조로 이루어진다.

| ID | 기능 | 설명 | 우선순위 |
|---|---|---|---|
| SCH-01 | 일정 자동 생성 | 근무정책 + 휴일 캘린더를 기반으로 직원×일자 단위 근무일정을 배치 생성 (예: 매월 말 익월분 생성) | 필수 |
| SCH-02 | 일정 개별 조정 | 특정 직원의 특정 일자 일정 변경 (휴무↔근무 전환, 시간 변경). 사유 필수, 이력 보존 | 필수 |
| SCH-03 | 일정 재생성 | 정책 변경·휴일 추가 시 미래 일정 재생성 (개별 조정분은 보존 여부 선택) | 필수 |
| SCH-04 | 시프트 지원 | 시프트(주간조/야간조 등) 정의 및 일정에 시프트 배정 — 데이터 구조만 우선 반영 | 선택 |

**근무일정이 답하는 질문**
- 오늘이 이 직원의 근무일인가, 휴무일인가?
- 근무일이라면 몇 시 출근 / 몇 시 퇴근 예정인가?
- → 이 기준이 있어야 지각(예정 출근시각 이후 체크인), 조퇴(예정 퇴근시각 이전 체크아웃), 결근(근무일인데 기록 없음)을 자동 판정할 수 있다.

**판정 규칙 (근무일정 × 이벤트 대조)**
| 판정 | 조건 |
|---|---|
| 정상 | 예정 출근시각 이내 체크인 + 예정 퇴근시각 이후 체크아웃 |
| 지각 | 예정 출근시각 + 유예시간(정책) 이후 체크인 |
| 조퇴 | 예정 퇴근시각 이전 체크아웃 (승인된 조퇴/반차 제외) |
| 결근 | 근무일인데 출퇴근 이벤트 없음 (승인된 휴가/출장 제외) |
| 자율출퇴근 | 지각/조퇴 판정 없음, 근무시간만 집계 |

### 4.4 출퇴근 관리

출퇴근은 하루 1쌍의 출/퇴근이 아니라 **이벤트 로그** 단위로 기록한다.

| ID | 기능 | 설명 | 우선순위 |
|---|---|---|---|
| ATT-01 | 체크 이벤트 기록 | 체크 유형: 출근 / 퇴근 / 외출 / 복귀 / 외근 시작·종료 / 출장 / 재택 시작·종료. 서버 시간 기준 저장 | 필수 |
| ATT-02 | 기록 메타데이터 | 이벤트마다 IP, 브라우저(User-Agent), 기기 정보 자동 수집. GPS는 정책 확정 시 선택 수집 | 필수 |
| ATT-03 | 기록 보정 신청 | 체크 누락·오류 시 사유(+첨부파일)와 함께 보정 신청 → 승인 후 반영. **원본 이벤트는 불변, 보정 이벤트를 별도 기록**하고 계산 시 보정본 우선 적용 | 필수 |
| ATT-04 | 오늘의 근태 현황 | 본인 이벤트 타임라인, 계산된 근무시간, 주간 누적 근무시간 표시 | 필수 |
| ATT-05 | IP 대역 제한 | 사내 IP 대역에서만 체크 허용 (정책으로 on/off, 재택 근무형태는 예외 처리) | 선택 |

> **개인정보 유의사항**: IP·기기 정보·GPS는 개인정보에 해당하므로 수집 항목·목적·보존 기간을 사내 고지하고 동의 절차를 거친다. GPS는 필요성이 확정되기 전까지 수집하지 않는다(기본 off).

### 4.5 연차/휴가 관리

#### 4.5.1 휴가 유형 (leave_types — 정책 데이터)
| 유형 | 단위 | 유급 여부 | 첨부파일 | 비고 |
|---|---|---|---|---|
| 연차 | 1일 / 0.5일(반차) | 유급 | — | 연차정책 기준 자동 발생 |
| 대체휴가 | 1일 / 0.5일 | 유급 | — | 휴일근무 승인 건에서 발생 |
| 경조휴가 | 일 단위 | 유급 | 선택 | 유형별 일수 정책 설정 |
| 병가 | 일 단위 | 정책 설정 | **필수 옵션** (진단서) | 첨부 필수 여부는 유형별 설정 |
| 공가 | 일 단위 | 유급 | 선택 (예비군 통지서 등) | |
| 무급휴가 | 일 단위 | 무급 | — | |

> 휴가 유형은 HR이 관리 화면에서 추가/수정하는 정책 데이터. 유형별로 `첨부파일 필수 여부`, `증빙 요구 조건`을 설정할 수 있다.

#### 4.5.2 연차정책 (leave_policies — 유형과 분리된 정책 테이블)
연차의 **발생·소멸 규칙**은 휴가 유형과 분리하여 정책으로 관리한다. 직원별/전사 기본값으로 배정 가능 (4.1.1 우선순위 규칙).

| 정책 항목 | 설정 값 예시 |
|---|---|
| 부여 기준 | 입사일 기준 / 회계연도 기준 |
| 발생 규칙 | 근로기준법 제60조 기본 (1년 미만 월 1일 최대 11일 / 1년 이상 15일 / 2년마다 +1일 최대 25일) |
| 사용기한 | 발생일로부터 1년 (설정 가능) |
| 이월 | 허용 여부, 이월 한도 |
| 소멸 | 기한 도래 시 자동 소멸 (소멸 내역 기록) |
| 촉진 | 촉진 알림 시점 (D-60/D-30 등), 대상 조건 |
| 사용 단위 | 1일 / 반차(0.5) / 반반차(0.25, 허용 시) |

- 고용형태별로 다른 정책 적용 가능 (예: 아르바이트는 별도 정책).
- 정책 변경은 적용 시작일 기준 버전 관리 — 기발생 연차에는 소급 적용하지 않는다.

#### 4.5.3 휴가 신청/승인
| ID | 기능 | 설명 | 우선순위 |
|---|---|---|---|
| LEV-01 | 휴가 신청 | 유형, 기간, 반차 여부, 사유, 첨부파일(유형별 필수 설정) 입력. 잔여일수 초과·**기간 중복 신청 시 차단** | 필수 |
| LEV-02 | 승인/반려 | 승인라인에 따른 다단계 승인. 반려 시 사유 필수. 결과 알림 발송 | 필수 |
| LEV-03 | 신청 취소 | 승인 전 자유 취소, 승인 후 시작일 전이면 취소 신청(승인 필요) | 필수 |
| LEV-04 | 잔여 연차 조회 | 발생 건별 발생/사용/잔여/소멸 예정 내역 조회 | 필수 |
| LEV-05 | 부서 휴가 캘린더 | 부서원 휴가 일정 월 캘린더 표시 (승인자 판단 지원) | 필수 |
| LEV-06 | 연차 수동 조정 | HR이 발생/차감 수동 조정 (사유 필수, 감사 로그 기록) | 필수 |

- 휴가 일수 계산 시 **근무일정 기준으로 휴무일(주말·휴일) 자동 제외**.
- 잔여 차감은 승인 시점 확정, 신청 시점에는 가차감(pending)으로 이중 신청 방지.
- 승인된 휴가는 해당 일자의 근태 판정에 자동 반영 (결근 아님).

### 4.6 초과근무/휴일근무
| ID | 기능 | 설명 | 우선순위 |
|---|---|---|---|
| OT-01 | 초과근무 신청 | 사전 신청 원칙 (일시, 예상 시간, 사유). 사후 신청 허용 여부는 정책 설정 | 필수 |
| OT-02 | 승인/반려 | 승인라인에 따른 승인. 승인된 건만 초과근무로 집계 | 필수 |

### 4.7 승인라인 관리

"다단계 확장 가능" 수준이 아니라, 처음부터 **승인라인 템플릿 + 단계** 구조로 설계한다.

| ID | 기능 | 설명 | 우선순위 |
|---|---|---|---|
| APV-01 | 승인라인 정의 | 승인라인(템플릿) 등록: 이름, 적용 대상(신청 유형), 단계 목록 | 필수 |
| APV-02 | 단계 정의 | 각 단계의 승인자 지정 방식: 특정인 / 직속 부서장 / 상위 부서장 / 특정 직책 | 필수 |
| APV-03 | 라인 배정 | 직원별/부서별 승인라인 배정 (4.1.1 우선순위 규칙) | 필수 |
| APV-04 | 승인 처리 | 순차 승인, 중간 반려 시 종료. 승인자 부재 시 대결(위임) 설정 | 필수 (대결은 선택) |
| APV-05 | 승인함 | 승인자별 대기 건 목록, 일괄 승인 | 필수 |

**예시**: `일반 휴가 라인` = 팀장(1단계) / `장기 휴가 라인` = 팀장 → 본부장 / `임원 라인` = 대표

- 승인 구조는 휴가·초과근무·근태 보정이 공통으로 사용하는 **독립 모듈**로 구현한다 (9장).

| 검증 | 내용 |
|---|---|
| 짝 없는 이벤트 | 출근만 있고 퇴근 없음 (또는 반대), 외출 후 복귀 없음 |

| 잔여 연차 음수 | 조정/취소 과정에서 잔여가 음수가 된 직원 |

### 4.11 통계/리포트

**기본 리포트**
- 개인별 월간 근태 요약 (근무일수, 지각/조퇴/결근, 초과근무 시간, 휴가 사용)
- 부서별/전사 기간별 집계

**HR 활용 리포트**
| 리포트 | 내용 |
|---|---|
| 월별 지각 현황 | 직원별/부서별 지각 횟수·시간 추이 |
| 부서별 연차 사용률 | 발생 대비 사용률, 부서 간 비교 |
| 연차 소멸 예정자 | 사용기한 임박(D-60/D-30) 직원 목록 — 촉진 대상 식별 |
| 초과근무 상위 직원 | 기간별 초과근무 누적 상위 목록, 52시간 근접자 |
| 근태 이상자 현황 | 지각/결근/짝 없는 이벤트 반복 직원 |
| 월별 출근율 | 부서별/전사 출근율 추이 |

- 전 리포트 Excel 다운로드 지원.

### 4.12 알림
| 이벤트 | 수신자 | 채널 (1단계) |
|---|---|---|
| 휴가/초과근무/보정 신청 | 해당 단계 승인자 | 시스템 내 알림 + 이메일 |
| 승인/반려 결과 | 신청자 | 시스템 내 알림 + 이메일 |
| 연차 사용기한 임박 (정책 설정 시점) | 본인, HR | 시스템 내 알림 + 이메일 |

| 마감 전 검증 이상 건 | HR | 시스템 내 알림 |

```
                      적용 범위: 전사/부서)

■ 근무일정 모듈
shifts               시프트 정의 (이름, 시작/종료 시각 — 교대근무 대비, 선택)
work_schedules       근무일정 (직원, 일자, 근무일 여부, 예정 출근/퇴근 시각,
                      shift_id?, 생성 근거: 정책 자동/수동 조정, 조정 사유)

■ 출퇴근 모듈
attendance_events    출퇴근 이벤트 (직원, 유형: 출근/퇴근/외출/복귀/외근/출장/재택,
                      발생 시각, IP, user_agent, 기기 정보, GPS(선택, 기본 미수집),
                      원본/보정 구분, 보정 원본 이벤트 참조)
attendance_corrections  보정 신청 (대상 일자, 변경 내용, 사유, 상태, 첨부)
※ 일 단위 근태 판정·근무시간은 저장하지 않음 — work_schedules × attendance_events로 계산

■ 휴가 모듈
leave_grants         연차/휴가 발생 (직원, 유형, 발생일, 일수, 사용기한, 발생 사유,
                      소멸 일수/소멸일)
leave_requests       휴가 신청 (기간, 일수, 상태: 대기/승인/반려/취소)
leave_usages         발생 건별 사용 매핑 (grant ↔ request, 선입선출 차감)

■ 초과근무 모듈
overtime_requests    초과근무/휴일근무 신청 (일시, 예상/실적 시간, 상태)

■ 승인 모듈 (독립 — 전 신청 유형 공통)
approval_lines       승인라인 템플릿 (이름, 적용 신청 유형)
approval_line_steps  단계 정의 (순번, 승인자 지정 방식: 특정인/직속 부서장/직책)
approvals            승인 인스턴스 (신청 유형 + 신청 ID, 현재 단계, 상태)
approval_step_records  단계별 처리 이력 (승인자, 처리 일시, 의견)

■ 지원 모듈
attachments          첨부파일 (참조 유형 + 참조 ID 범용 구조, 파일 메타, 접근 권한)
notifications        알림
audit_logs           감사 로그 (대상, 변경 전/후 값(JSON), 사유, 변경자, 일시, IP)
monthly_closings     월 마감 상태 (대상 월, 상태, 실행자, 검증 결과)
closing_snapshots    마감 스냅샷 (직원×월: 근무시간, 지각/조퇴/결근 횟수,
                      초과근무 시간, 휴가 사용 집계 — 마감 후 불변, 급여 근거)
```

**설계 원칙 요약**
- 파생값 미저장: 잔여 연차 = `leave_grants − leave_usages` 계산, 근무시간·지각 = `work_schedules × attendance_events` 계산. 성능 이슈 시 캐시(무효화 가능한) 도입, 원본 우선.
- 마감 스냅샷 예외: 급여 근거는 마감 시점 값으로 고정 (1.3장).
- 원본 불변: 출퇴근 이벤트·승인 이력은 수정하지 않고 보정/취소 레코드를 추가.
- 근태 도메인 테이블은 `users`가 아닌 `employees`(사번)를 참조 → ERP 병합 시 계정 체계 변경에 영향 없음.

## 8. 기술 스택 (제안)

> daolERP 본체의 스택이 아직 확정되지 않았으므로, 아래는 제안이며 **ERP 본체 스택 결정과 함께 확정**해야 한다. 근태 시스템이 사실상 ERP의 첫 모듈이 되므로, 여기서 정한 스택이 ERP 표준이 될 가능성이 높다.

| 구분 | 제안 | 비고 |
|---|---|---|
| 백엔드 | Spring Boot (Java/Kotlin) 또는 NestJS (TypeScript) | 팀 숙련도에 따라 선택 |
| 프론트엔드 | React + TypeScript | 반응형 웹 |
| DB | PostgreSQL | 트랜잭션·정합성 중시. before/after JSON 로그에 JSONB 활용 |
| 인증 | 1단계: 자체 세션/JWT → 2단계: ERP 통합 인증(SSO) | |
| 파일 저장소 | 로컬 볼륨 → 2단계 오브젝트 스토리지 검토 | 첨부파일 |
| 배치 | 스케줄러 (근무일정 생성, 연차 발생/소멸, 촉진 알림, 52시간 집계) | |
| 배포 | Docker 기반 | ERP 병합 시 동일 인프라 사용 |

## 9. 모듈 구조 및 ERP 병합 전략

### 9.1 모듈 구조
병합 시 영향도를 최소화하기 위해 도메인별 독립 모듈로 구성한다. 모듈 간 참조는 인터페이스를 통해서만 한다.

```
common      사용자/직원/조직/권한  ← 2단계에서 ERP 공통 모듈로 교체되는 유일한 부분
policy      근무정책/연차정책/휴가유형/휴일
schedule    근무일정/시프트
attendance  출퇴근 이벤트/보정/근태 계산
leave       연차 발생/신청/잔여
overtime    초과근무
approval    승인라인/승인 처리      ← 근태 외 ERP 타 모듈(전자결재 등)에서도 재사용 가능
attachment  첨부파일               ← 범용 구조, ERP 전반 재사용 가능
notification 알림                  ← 2단계에서 ERP 공통 알림으로 통합
closing     월 마감/스냅샷
report      통계/리포트
```

### 9.2 병합 절차 (2단계)
- ERP 공통 모듈(사용자/조직/권한)이 준비되면 `common` 모듈 구현체만 ERP 것으로 교체
- 직원/부서 데이터 매핑 마이그레이션 (사번 기준)
- 나머지 모듈의 테이블은 데이터 이관 없이 스키마째 이동 (동일 DB라면 그대로 사용)
- `approval`/`attachment`/`notification`은 ERP 공통 서비스로 승격 후보 — 병합 시점에 ERP 표준으로 채택 여부 결정
- 화면은 ERP 셸(공통 레이아웃/메뉴) 안에 근태 메뉴로 편입

### 9.3 검증 기준 (병합 GO/NO-GO)
독립 운영 기간 중 아래 충족 시 병합 진행:
- 1회 이상 월 마감을 실데이터로 무리 없이 완료 (자동 검증 → 마감 → 스냅샷 → 내보내기 전체 사이클)
- 연차 자동 발생 결과가 수기 계산과 전수 일치
- 근태 판정(지각/조퇴/결근)이 실제와 일치하는지 표본 대조 완료
- 사용자 불편 사항(버그/개선) 처리 완료

## 10. 개발 단계 및 일정 (안)

기준 데이터가 먼저 만들어지는 순서로 진행한다: **조직 → 정책 → 일정 → 승인 골격 → 출퇴근 → 휴가 → 리포트/마감**.

| 단계 | 기간(안) | 내용 |
|---|---|---|
| M1 | 2주 | 프로젝트 셋업, 공통 도메인(사용자/직원(확장 필드)/조직/직급·직책/권한/로그인), 화면 골격 |
| M2 | 2주 | 근무정책·연차정책·휴가유형·휴일 관리, 정책 배정(직원/부서/기본값) |
| M3 | 2주 | 근무일정: 자동 생성 배치, 개별 조정, 재생성 + **승인 모듈 골격**(라인/단계/승인함) |
| M4 | 2주 | 출퇴근: 이벤트 기록(메타데이터 포함), 보정 신청·승인, 근태 판정 계산, 첨부파일 |
| M5 | 3주 | 휴가: 연차 자동 발생/소멸 배치, 신청·승인, 잔여 관리, 부서 캘린더, 촉진 알림 |
| M6 | 2주 | 초과근무, 52시간 모니터링, 알림 통합 |
| M7 | 2주 | 월 마감(자동 검증 + 스냅샷 + 내보내기), 통계/리포트 |
| M8 | 1주 | 통합 테스트, 실데이터 검증(연차 계산 전수 대조, 근태 판정 표본 대조), 배포 |
| 운영 검증 | 1~3개월 | 실사용 및 피드백 반영 → 병합 GO/NO-GO 판단 |

> 총 개발 약 16주 + 운영 검증 (v0.1의 13주 대비 +3주 — 근무일정·승인라인·정책 구조 확장 반영). 인력 규모에 따라 조정.
