# 매입 3축 연결 설계안 (Purchase 3-Axis Linkage)

> 목표(사장님): **ERP 주문의 매입내역 · 매입 세금계산서 · 통장 지급내역** 세 가지가 서로 연결되어,
> "무엇을 사기로 했고(ERP) → 실제 매입 증빙을 받고(세금계산서) → 돈을 지급했다(통장)"가 하나로 추적되어야 한다.

작성일: 2026-07-08 · 상태: **설계안 (구현 전 승인 대기)**

---

## 1. 세 축의 성격 (먼저 정확히 구분)

| 축 | 데이터 | 단위 | 성격 |
|---|---|---|---|
| ① ERP 매입내역 | erp_order_items.purchase_* | **주문 품목별** (건별로 잘게) | "이 상품을 이 매입처에서 얼마에 사서 고객에게 보냈다" — 매출원가의 근거 |
| ② 매입 세금계산서 | tax_invoices (purchase) | **거래처·월 합계** (뭉쳐서) | 세금계산서 = 세무 증빙. 매입처가 월별로 몰아서 발행하는 경우가 많음 |
| ③ 통장 지급 | transactions (출금) | **지급 건별** (분할 지급도) | 실제 현금 유출. 미지급금을 갚는 행위 |

**핵심 난점 (실데이터 진단으로 확인됨):**
- ①은 품목 단위, ②는 월 합계 단위 → **건 대 건 매칭이 원천적으로 불가능**
- 거래처×월로 묶어 비교해도 금액이 정확히 안 맞음 (양쪽 있는 460조합 중 3% 이내 근사는 36조합뿐)
  - 이유: 매입 시점과 계산서 발행 시점의 **시차**(월 걸침), 매입단가와 계산서금액의 **차이**, ERP엔 있으나 계산서는 별도 처리되는 건
- 상위 매입처도 ERP 매입액 > 세금계산서 공급가인 경향 (예: 다온글로비스 ERP 4.7억 vs 세계 3.5억)

**결론: "완전 자동 건별 매칭"은 목표가 아니다.** 대신 **거래처 단위로 세 축을 나란히 보여주고 차이를 드러내는 대사(對査) 구조**가 맞다.

## 2. 연결 방식 — 축 조합별

### ②↔③ 매입 세금계산서 ↔ 통장 지급 (이미 구현됨, 확장 필요)
- 현재: 세금계산서 화면에서 통장 출금과 결제매칭 → 미지급금 상계 (파인갤러리 사례). **66건 연결**.
- 방식: **금액 기반 매칭** (계산서 합계 = 지급액, 분할지급 합산 지원)
- 확장: 선택 자동매칭(거래처+금액+기간)으로 미매칭 2,251건을 후보 추천 → 사용자 확정
- 회계: 계산서=비용 발생(상품매입/미지급금), 통장=미지급금 상계 → **이중계상 없음** (이미 정립)

### ①↔② ERP 매입 ↔ 세금계산서 (신규 — 핵심 공백)
- 건별 매칭 불가 → **거래처 × 기간 집계 대사**로 설계
- 한 거래처에 대해: `ERP 매입액 합계` vs `세금계산서 공급가 합계`를 기간별로 나란히 표시
- 차이를 3가지로 자동 분류:
  - **정상 범위** (오차 ±3% 이내): 매칭 완료로 간주
  - **시차 의심** (전월/익월 합산 시 일치): 계산서 발행 시점 차이 안내
  - **차이 큼**: 계산서 미수령 / 매입단가 상이 / ERP 미입력 → 사용자 검토
- 연결 저장: 건별 링크가 아니라 **거래처×기간 대사 확정** 기록 (누가·언제 확인했는지)

### ①↔③ ERP 매입 ↔ 통장 (간접 — ②를 경유)
- ERP 매입 → (세금계산서) → 통장 지급의 **경유 연결**. 직접 링크는 만들지 않음.
- 세금계산서를 안 받는 매입처(현금 직매입 등)만 예외적으로 ERP↔통장 직접 대사 대상

## 3. 통합 화면 구성 (제안)

**매입처 3축 대사 대시보드** — 거래처 하나를 고르면 한 화면에:

```
[거래처: (주)다온글로비스]              기간: 2025-01 ~ 2025-12
┌─────────────────────────────────────────────────────────┐
│ ① ERP 매입   473,680,450   (주문 품목 N건)                │
│ ② 세금계산서  348,882,404   (계산서 M건, 계정확정 M건)      │  ← ①과 차이 자동 표시
│ ③ 통장 지급   340,000,000   (지급 K건, 계산서 연결 K건)     │  ← ②와 차이(미지급 잔액)
├─────────────────────────────────────────────────────────┤
│ 미지급금 잔액: 8,882,404   |   ERP-세계 차이: 124,798,046  │
└─────────────────────────────────────────────────────────┘
```
- 각 숫자 클릭 → 해당 건 목록 → 원본까지 드릴다운 (기존 추적성 재사용)
- 차이가 큰 거래처를 상단에 정렬 → "확인이 필요한 매입처"부터 처리

## 4. 자동 / 수동 경계 (정책 §4 준수 — 추천만, 확정은 사용자)

| 작업 | 자동 | 사용자 |
|---|---|---|
| ②↔③ 결제 후보 제시 | 금액·거래처·기간으로 후보 추천 | 매칭 확정 |
| ①↔② 대사 계산·차이 분류 | 집계·차이·시차 자동 판정 | "확인함" 확정, 예외 판단 |
| 계정과목·분개 | 확정 시 자동 생성 (기존) | 계정 확정 |

**절대 안 하는 것**: ERP 매입과 세금계산서를 시스템이 임의로 "같다"고 확정하지 않는다. 금액 차이가 정상이어도 최종 확인은 사용자.

## 5. 구현 로드맵 (단계별 — 각 단계 검증 보고서 포함)

1. **①↔② 대사 API + 거래처별 대사표** (거래처×기간 ERP매입 vs 세계, 차이 분류)
2. **매입처 3축 대사 대시보드 화면** (①②③ 한 줄 요약 + 드릴다운)
3. **②↔③ 자동매칭 확장** (미매칭 계산서에 통장 지급 후보 추천 — 매출 수금매칭 로직 재사용)
4. **대사 확정 이력** (거래처×기간 "확인함" 저장 + 재대사 시 변경 감지)
5. 실데이터 검증 — 상위 매입처부터 3축 차이 원인 규명 (2025 더존 대사와 연계)

## 6. 이 설계가 지키는 원칙 (회계정책 §3·§4·§6)
- **역할 분리 유지**: 비용은 세금계산서가 1회 계상, 통장은 지급(상계). ERP 매입은 매출원가 근거(손익엔 ERP 원가). 3축 연결이 이 구조를 바꾸지 않는다 — **추적을 더할 뿐 이중계상을 만들지 않는다.**
- **추천만**: 대사·후보는 자동, 확정은 사용자.
- **추적성**: 모든 숫자는 원본까지 드릴다운.
