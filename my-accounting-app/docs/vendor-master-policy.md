# 거래처 마스터 정책 (Vendor Master Policy)

> 이 문서는 daolERP의 거래처 데이터 관리 기준이다.
> 통장 자동분류·카드 자동분개·세금계산서 자동분개·거래처 원장 등 모든 기능은 이 정책을 따른다.
> 구현보다 정책이 우선한다 — 정책과 코드가 어긋나면 정책을 기준으로 코드를 고친다.

---

## 1. 거래처 마스터의 개념

**`vendors` 테이블이 ERP 전체의 유일한 거래처 원본(Single Source of Truth)이다.**

```
                    ┌─────────────────────┐
                    │   vendors (마스터)    │
                    │  대표명 · 사업자번호   │
                    │  별칭 · 카드번호      │
                    └──────────┬──────────┘
        ┌──────────┬──────────┼──────────┬──────────┐
        ▼          ▼          ▼          ▼          ▼
   ERP 주문     통장 거래   세금계산서   카드매출/    분개
 (별칭 경유)  (vendor_id)  (vendor_id)  법인카드   (journal_lines
                                       (vendor_id)   .vendor_id)
        └──────────┴──────────┴──────────┴──────────┘
                            ▼
                      거래처별 원장
```

이미 9개 테이블이 vendors를 참조한다(transactions·journal_lines·tax_invoices·card_sales·
cash_receipts·erp_vendor_aliases·vendor_ledger_entries·card_accounts·vendor_opening_balances).
**새 기능이 거래처 개념을 쓸 때는 반드시 vendors.id를 참조한다. 이름 문자열 저장 금지.**

- ERP 매출처/매입처 이름(`erp_vendor_aliases`)은 마스터가 아니라 **마스터를 가리키는 별칭 계층**이다.
  모든 ERP 별칭은 궁극적으로 vendor_id에 연결되는 것을 목표로 한다.
- 거래처의 역할 속성은 마스터의 플래그로 구분한다:
  `type`(customer/vendor/both) · `is_card_company`(카드사 정산용) · `is_related_party`(관계사, 도입 예정).

## 2. 대표명과 별칭

| 개념 | 정의 | 저장 위치 |
|---|---|---|
| **대표명** | 거래처를 대표하는 단 하나의 이름. 화면·원장·보고서에 표시 | `vendors.name` |
| **별칭** | 대표명 외에 이 거래처를 가리키는 모든 표기 | `vendors.match_aliases[]` |
| **ERP 이름** | ERP 파일에 등장한 매출처/매입처 문자열 | `erp_vendor_aliases.erp_name` → vendor_id |
| **카드번호** | 이 거래처가 결제에 쓰는 법인카드 번호 | `vendors.card_numbers[]` |

예:
```
대표명: 올푸름
별칭:   주식회사 올푸름 / (주)올푸름 / ㈜올푸름 / OLPUREUM
```

### 대표명 선정 기준 (우선순위)
1. **사업자등록 상호** (세금계산서에 찍히는 공식 상호) — 있으면 최우선
2. 공식 상호가 없거나 불명확하면 **실무에서 가장 많이 쓰는 이름**
3. 표기는 간결형을 선호: 법인격(주식회사·(주)) 은 공식 상호가 아니면 생략 가능
4. 은행 지점·부서 거래처는 `은행명 + 지점/부서명` 형식으로 통일 (예: `하나은행 교대역금융센터`)

대표명 변경은 언제든 가능하며, 이전 대표명은 자동으로 별칭에 보존한다.

## 3. 이름 정규화 규칙 (매칭 전 공통 적용)

두 이름이 같은 거래처인지 비교할 때는 아래를 무시한다:
- 법인격 표기: 주식회사·(주)·㈜·(유)·유한회사·(재)·(사)·(합)
- 공백·특수문자(`- _ . , · ( ) [ ] ' " ! @ # & + / \`)·대소문자
- 관용 동의어: `종합금융센터 = 금융센터`, `KEB하나은행 = 하나은행` (동의어 사전은 추가 관리)

## 4. 자동 매칭 우선순위

원천 데이터(통장 적요·세금계산서 상호·ERP 이름·카드번호)에서 거래처를 찾을 때:

| 순위 | 기준 | 결과 처리 |
|---|---|---|
| ① | **사업자번호 일치** | 자동 연결 |
| ② | 정산 전용 패턴 + `is_card_company` (카드정산 입금) | 자동 추천 |
| ③ | **카드번호 일치** (`card_numbers`) | 자동 연결 |
| ④ | 대표명 **정확 일치** (정규화 후) | 자동 연결/추천 |
| ⑤ | **별칭 일치** (정규화 후) | 자동 연결/추천 |
| ⑥ | 부분 문자열 일치 | 추천 (검토 표시) |
| ⑦ | 짧은 이름(2~3자) 부분 일치 | **검토 필요** — 자동 연결 금지 |

- **부분 문자열 매칭 최소 조건**: 정규화 후 **4자 이상**이어야 하며, **후보가 하나일 때만** 추천한다.
- 모든 단계 공통: **후보가 단 하나일 때만** 자동 연결/추천한다. 복수 후보 → 사용자에게 **선택 요청**.
- 관계사(`is_related_party`)는 어떤 경우에도 계정과목 자동분류하지 않는다 (거래처 태깅만).

### 신뢰도(Confidence) — 내부 전용
- 신뢰도 수치(0.9/0.7, %, 별점 등)는 **내부 로직·로그에서만** 사용한다.
- 화면에는 세 가지 상태만 노출한다: **자동 추천 / 검토 필요 / 선택 요청**.

## 5. 중복 예방 (병합보다 우선)

**원칙: 중복은 만들어진 뒤 병합하는 것보다, 처음부터 안 만드는 것이 우선이다.**

1. **거래처 신규 등록 시** — 정규화 이름·사업자번호로 기존 거래처를 검사하고, 유사 거래처가 있으면:
   > "기존 거래처 '○○○'와(과) 매우 유사합니다. 기존 거래처를 사용하시겠습니까?"
   사용자가 "새로 만들기"를 명시적으로 선택해야만 생성한다.
2. **ERP 임포트 시** — 새 ERP 이름이 등장하면:
   - 기존 별칭·저장된 변형 이름과 정규화 일치 → **기존 별칭/거래처로 자동 연결** (새 별칭 생성 안 함)
   - 유사하지만 확실치 않음 → 별칭은 만들되 **검토 큐**에 표시
3. **자동 생성 최소화** — 시스템이 거래처를 자동 생성하는 경우는 카드사(054)처럼
   명시적으로 정의된 케이스로 한정하고, 문서에 기록한다.

## 5-1. 거래처 상태 (Status)

거래처는 생애주기 상태를 가진다. 초기에는 active/merged만 사용하더라도 상태 체계는 처음부터 둔다.

| 상태 | 의미 |
|---|---|
| `active` | 정상 사용 중 (기본값) |
| `inactive` | 거래 중단 — 목록·추천에서 숨기되 이력은 유지 |
| `merged` | 다른 거래처로 병합됨 — `merged_into_id`로 대표를 가리킴. 신규 연결 금지 |
| `archived` | 보관 — 과거 데이터 조회 전용 |

- 매칭·추천 엔진은 `active`만 대상으로 한다. (`merged`는 대표로 자동 치환)
- 기존 `is_active`는 호환을 위해 유지하되, status가 기준이다.

## 6. 병합 정책

- **자동 병합 금지.** 시스템은 중복 후보만 제시하고, 병합은 반드시 사용자가 승인한다.
- 병합 실행 시:
  1. 사용자가 그룹에서 **대표를 선택**
  2. 흡수되는 거래처/별칭의 모든 참조(주문·통장·세금계산서·카드·분개라인·기초잔액)를 대표로 이전
  3. 흡수된 이름은 대표의 **별칭으로 보존** (이후 같은 이름이 와도 대표로 인식 — 학습)
  4. 흡수된 거래처 행은 삭제하지 않고 **status='merged' + merged_into_id**로 남긴다
- **병합 이력(merge log) 필수**: 일시 · 수행자 · 대표 id/이름 · 흡수 id/이름 · 이전된 참조 건수.
- **복구(undo)**: 이력에 이전 건수·범위를 남겨 장기적으로 역이전 복구를 지원한다.
  (v1은 "이력 완전 기록"까지를 필수로 하고, 복구 기능은 이력 기반으로 후속 구현)

## 7. 학습 루프 — 사용자의 판단이 데이터가 된다

사용자의 확정 행동은 모두 별칭/매칭 데이터로 축적한다:

| 사용자 행동 | 시스템 학습 |
|---|---|
| "이 둘은 같은 거래처" 병합 승인 | 흡수 이름 → 별칭 저장 → 다음 임포트부터 자동 연결 |
| 통장/세금계산서에서 거래처 수동 지정 | 해당 표기 → 별칭 후보로 축적 |
| 카드번호가 포함된 주문 확인 | 카드번호 → `card_numbers` 저장 → 카드매출 자동 고객 매칭 |
| 계정과목 확정 반복 | (거래처별 확정 이력 → 추천 정확도 향상, 분류 정책 문서 참조) |

시간이 지날수록 자동 인식률이 올라가는 구조를 목표로 한다. 학습된 값도 언제나 사용자가
별칭 관리 화면에서 수정·삭제할 수 있다.

## 7-1. 거래처 기본계정 (default_account_id)

거래처에 저장하는 "이 거래처는 기본적으로 이 계정" 정보. 시간이 지날수록 추천의
기준 데이터가 되므로 **변경 주체를 사용자로 고정**한다.

| 원칙 | 내용 |
|---|---|
| 자동 변경 금지 | 시스템·배치·학습 로직이 기본계정을 자동으로 바꾸지 않는다 |
| 변경 경로 2가지뿐 | ① 분류 화면의 "기본계정 저장" 체크(사용자 명시 행위) ② 거래처 관리 화면 직접 지정 |
| 백필은 빈 값만 | 일괄 백필(세무사 자료 등)은 기본계정이 비어 있는 거래처에만, 과반 계정만 저장 |
| 이력이 우선 | 추천 우선순위는 **확정 이력 과반 > 기본계정 > 키워드** — 기본계정이 낡아도 실제 확정 이력이 추천을 교정한다 |
| 추천 전용 | 기본계정은 추천 재료일 뿐, 어떤 경우에도 자동 확정·자동 분개의 근거가 되지 않는다 |

## 8. 모듈별 연결 흐름 (요약)

| 모듈 | 거래처 연결 방식 | 비고 |
|---|---|---|
| ERP 주문 | ERP 이름 → `erp_vendor_aliases` → vendor_id | 별칭 계층 경유 |
| 통장 거래 | 자동 매칭(§4) → `transactions.vendor_id` | 분류 확정 시 분개 라인에 전파 |
| 세금계산서 | 사업자번호 우선 매칭 → `tax_invoices.vendor_id` | 채권/채무 라인에 태깅 |
| 카드매출 | 카드번호(`card_numbers`) → `card_sales.vendor_id` = **고객**<br>매입사 정규화 → 분개 매출채권 라인 = **카드사** | 고객 축과 정산(카드사) 축 분리 |
| 법인카드 | `card_accounts.vendor_id` = 카드사 (미지급금 상대처) | |
| 분개/원장 | `journal_lines.vendor_id` → 거래처별 원장 | 마스터 병합 시 원장도 자동 합산 |

## 9. 구현 로드맵

- **Phase 1 (이번)**: 병합 기능(이력 포함) + 중복 후보 제시 화면 + ERP 임포트 변형 이름 자동 연결
- **Phase 2**: 신규 등록/임포트 시 유사 거래처 경고 UX, 동의어 사전 관리
- **Phase 3**: 수동 지정 → 별칭 자동 축적, 병합 복구(undo)

## 10. 운영 철학

> **거래처 마스터는 사용자가 관리하는 자산이다.**
>
> ERP는 거래처를 자동으로 생성하거나 병합하는 시스템이 아니라,
> **사용자의 판단을 축적하여 점점 더 정확하게 연결을 도와주는 시스템**이다.
>
> 시간이 지날수록 별칭·카드번호·ERP 이름·확정 이력이 누적되어
> 자동 연결률은 높아지지만, **최종 판단 권한은 항상 사용자에게 있다.**

앞으로 AI 추천을 포함한 어떤 자동화 기능을 추가하더라도 이 철학을 벗어나지 않는다.
새 기능은 구현 전에 이 문서(Vendor Master Policy)와 충돌하지 않는지 먼저 확인한다.
