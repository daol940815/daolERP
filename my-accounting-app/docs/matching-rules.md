# 매칭 규칙 통합 문서

모든 매칭 도구가 따르는 공통 규칙. 코드 기준은 `lib/matching-rules.ts` — 상수·판별 함수를
반드시 그 모듈에서 가져다 쓴다. 이 문서와 코드가 어긋나면 코드를 고치든 문서를 고치든 함께 바꾼다.

## 업무 원칙 (다올커머스)

1. **계산서 발행 후 지급/수금이 원칙.** 매출은 발행 후 입금받고, 매입은 발행 후 지급한다.
   발행 전 거래(선지급/선입금)는 예외이며, **자동으로 연결하지 않고 사용자가 확인**한다.
2. **선지급 관행 거래처는 이력에서 학습.** 고정지출을 미리 지급하고 계산서를 나중에 받는
   거래처(파인갤러리 유형)는 사용자의 확정 이력으로 판정한다 — 발행 전 지급 확정이
   2건 이상이고 전체 확정의 50% 이상이면 관행으로 본다. 하드코딩 예외 목록은 두지 않는다.
3. **금액 조건은 어디서도 완화하지 않는다.** 정확 일치 + 잔여금액(기연결 차감) 검사.
4. **추천만 한다 — 확정은 사용자.** 유일하게 자동 연결하는 도구는 자동매칭(auto-match)뿐이고,
   그마저 아래 안전장치를 전부 통과한 건만 연결한다.
5. 매칭 판정은 날짜 단위(상대 자료가 일 단위). 시각은 표시용.

## 상대 판별 (4신호) — `makeIdentityScorer`

거래처 태깅(+3) / 사업자번호 포함(+2) / 상호명 포함(+2) / 학습 별칭 포함(+2). 0점 = 무관.

## 도구별 규칙표

| 도구 | 역할 | 날짜 규칙 |
|---|---|---|
| 자동매칭 `api/tax-invoices/auto-match` | **자동 연결** | 발행 전 제외. 후보 여럿이면: 동일 거래처 + 최근접 35일 이내 + 2등과 15일 이상 격차. 발행 전 거래가 (최근접+7일)보다 가까우면 애매 → 수동. 발행일 오름차순 처리. 선지급 관행 거래처도 예외 없이 발행 전은 연결 안 함. |
| 개별 매칭 모달 `api/tax-invoices/[id]/match-candidates` | 후보 제시 | 점수 → 발행 후 우선(직전 7일 유예, 관행 거래처 31일) → 최근접. 발행 전 후보는 숨기지 않고 "발행 전 N일" 배지. |
| 합계 매칭 `api/tax-invoices/match-sum` | 후보 제시 | 위와 동일 (기준일 = 가장 늦게 발행된 계산서). |
| 지급연결후보 `lib/purchase-payment-match` | 후보 제시 (1:1·분할·역합산) | 발행 후 0~120일, 분할 조합은 30일 이내 우선. 관행 거래처는 발행 전 31일까지 포함. |
| 거래처 자동매칭 `api/transactions/match-vendors` | 거래 태깅 | 날짜 무관 (상호·별칭·사업자번호 유일 일치 시 태깅). |
| ERP 수금 / 카드매출 | 별도 사이클 | 각 도구 문서 참조 (카드번호 ±7일 유일 대응 등). |

## 상수 (lib/matching-rules.ts `MATCH_RULES`)

| 상수 | 값 | 의미 |
|---|---|---|
| AUTO_MARGIN_DAYS | 15 | 자동매칭: 최근접이 2등보다 이만큼 가까워야 유일 |
| AUTO_MAX_LAG_DAYS | 35 | 자동매칭: 최근접이 발행 후 이 일수 이내 |
| AUTO_PRE_GUARD_DAYS | 7 | 자동매칭: 발행 전 거래가 이만큼 근접하면 수동 |
| MANUAL_PRE_GRACE_DAYS | 7 | 수동 후보: 발행 직전 유예(월말 정산 관행) |
| PREPAY_PRE_GRACE_DAYS | 31 | 선지급 관행 거래처의 유예/탐색 확장 |
| SPLIT_NEAR_WINDOW_DAYS | 30 | 분할 조합 1차 탐색 창 |
| SPLIT_MAX_WINDOW_DAYS | 120 | 분할/합산 최대 탐색 창 |
| PREPAY_MIN_COUNT / RATIO | 2 / 0.5 | 선지급 관행 판정 기준 |
