# 회계 정책 (Accounting Policy)

> daolERP의 회계 숫자가 **어떻게 만들어지고, 왜 그렇게 만드는지**에 대한 상위 원칙.
> 새 기능은 구현 전에 이 문서와 충돌하지 않는지 먼저 확인한다.
> (기술 세부는 `journal-design.md`, 거래처는 `vendor-master-policy.md` 참조)

목표: **"처리하면 회계가 자동 생성"** — 업무 데이터를 올리고 확정하면 분개·원장·손익이
자동으로 만들어지고, 그 결과가 더존 총괄원장과 대사 가능한 수준으로 정확해야 한다.

---

## 1. 신뢰의 단일 원천 — 분개 (Journal)

모든 장부 숫자(원장·손익·채권채무)는 **분개(journal_entries/lines)에서 나온다.**

- 분개는 원천(source_type + source_id) 기준 **멱등** — 같은 원천을 재확정하면 분개가
  교체되고, 확정을 해제하면 분개도 제거된다. 장부와 원천이 어긋난 채 남지 않는다.
- 대차 균형은 저장 시점에 RPC(post_journal)가 강제한다. 불균형 분개는 존재할 수 없다.
- 손익현황은 분개를 계정별로 동적 집계한다. 특정 계정을 하드코딩하지 않으므로
  새 계정·새 원천(법인카드 등)이 생겨도 자동 반영된다. (2026-07 분개 기반 전환)

## 2. 매출·매출원가는 ERP 기준 (2026-07 사장님 결정)

| 항목 | 기준 | 이유 |
|---|---|---|
| 매출 | ERP 주문 (취소·VIP·선결제 제외) | 실제 영업활동의 원천 데이터. 세금계산서보다 완전함 |
| 매출원가 | ERP 주문의 품목 원가 | 판매 시점 원가라 매출과 같은 달에 대응(수익-비용 대응 원칙). 재고 개념을 자동 흡수 |

**손익 집계에서 제외하는 역할분리 계정** — 분개는 유지하되 손익에선 ERP 수치가 대신한다:

| 계정 | 제외 이유 | 그 분개의 역할 |
|---|---|---|
| 4001 매출 | ERP 매출이 대신 | 매출채권 원장·원장대사용 |
| 5001 매출원가 / 5002 상품매입 | ERP 원가가 대신 | 미지급금(카드사·매입처) 원장·부가세대급금용 |
| 4004 외상매출금(income) | legacy 오분류 계정 | 비활성화 예정 |

ERP 원가와 실제 매입액(세계+카드 상품매입)의 차이는 **재고 증감**으로, 더존 대사 시
상품 계정 차이를 설명하는 자료가 된다.

## 3. 세 축의 역할 분리 — 같은 돈이 두 번 잡히지 않는 구조

| 축 | 원천 | 만드는 분개 | 손익 기여 |
|---|---|---|---|
| 매출 축 | ERP 주문 / 카드매출 / 매출 세계 | 카드매출: (차)1101/(대)4001+2003 · 세계: (차)1101/(대)4001+2003 | 손익 매출은 ERP만. 4001 분개는 채권 관리용 |
| 정산·수금 축 | 통장 입금 (카드사 정산, 세계 매칭 수금) | (차)예금/(대)1101 — **채권 상계, 수익 아님** | 없음 (이미 매출로 잡힌 돈의 회수) |
| 비용·지급 축 | 매입 세계 / 법인카드 / 통장 출금 | 발생 시점: (차)비용/(대)2001 · 지급 시점: (차)2001/(대)예금 — **채무 상계, 비용 아님** | 비용은 발생 시점 1회만 |

핵심 규칙: **발생(비용·수익)과 결제(입출금)를 구분한다.** 결제는 채권·채무의 상계일 뿐
손익에 다시 잡히지 않는다. 통장 분류 도구가 정산·카드대금·세계매칭 유형을 자동으로
갈라내는 이유가 이것이다.

부속 규칙:
- 계좌 간 이체는 분개(손익) 대상이 아니다 — 이체쌍 연결로 제외.
- 카드 취소는 역분개로 상쇄, 승인 거절은 전기 금지, 부가세는 1102/2003으로 분리.
- 매출처(고객) 축과 매입사(카드사 정산) 축은 다른 축이다 — 예: 하나은행 지점(고객)과
  하나카드(정산사)를 혼동하지 않는다.

## 4. 자동분류는 추천까지만 — 확정은 항상 사용자

- 분류기·학습·시드(세무사 자료 등)는 `suggested_account_id`(추천)만 만든다.
  **분개를 만드는 확정(confirmed)은 사용자 행위로만 발생한다.**
- 신뢰도 수치는 내부 전용 — 화면엔 자동추천/검토필요/선택요청 3단계만 노출한다.
- 추천 우선순위: **사용자 확정 이력 과반 > 거래처 기본계정 > 키워드**.
  사용자가 확정할수록 추천이 좋아지는 학습 루프가 기본 구조다.
- 외부 확정 자료(세무사 더존 장부 등)를 일괄 확정으로 반영할 때도
  **기존 사용자 확정값은 절대 덮어쓰지 않는다** (사장님 판단 우선).

## 5. 이중계상 방어선

1. 구조적 방어: §3의 역할 분리 (발생 1회 + 결제는 상계).
2. 도구적 방어: 통장 분류의 유형 자동 분해 + 상계 계정 프리셋.
3. 상설 검사: 이중계상 검사 리포트(3방향 교차 — 세계↔통장 확정, 통장↔세계 유사,
   카드↔세계 병행발행)를 분류 작업 후 수시 실행. **0건 유지가 손익 신뢰의 방어선.**

## 6. 추적성 (Drill-down)

숫자는 근거를 설명할 수 있어야 한다. 목표 경로:

```
월별 손익 → 계정별 원장 → 분개 → 원본 레코드(카드/세계/통장) → (가능한 경우) 업로드 이력
```

모든 분개가 source_type/source_id를 가지므로 이 경로는 데이터로 보장된다.
새 집계 화면을 만들 때는 "클릭해서 내려갈 수 있는가"를 기본 요건으로 한다.

## 7. 운영 원칙 — 구현보다 검증

새 회계 기능은 반드시 다음 사이클로 진행한다:

```
설계 보고 → 승인 → 구현 → 실데이터 검증 → 검증 보고서(적용/실패 건수·차이 분석) → 운영 적용
```

- 대량 데이터 작업은 실행 전 드라이런/시뮬레이션 결과를 먼저 보고한다.
- 데이터를 바꾸는 작업은 항상 검증 쿼리로 결과를 확인하고 수치와 함께 보고한다.
- 정확도 이슈는 "오차 0 또는 원인 전부 규명"이 기준이다 (2025 상품매출 대사 방식).
